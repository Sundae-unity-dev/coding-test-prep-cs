직무

QA 직무와 채용

QA도 직무에 따라 코딩 테스트와 면접에서 보는 것이 달라요. 내 목표 직무를 먼저 정해요.

수동 QA (Manual QA)

명세와 시나리오 기반으로 직접 테스트하고 탐색적 테스트로 결함을 찾아요. 버그 리포트와 테스트 케이스 작성이 핵심이에요.

테스트 이론 중심

자동화 QA / SDET

테스트를 코드로 작성해요. UI 자동화(Selenium, Playwright), API 테스트, 단위 테스트, CI 연동을 다뤄요. 프로그래밍 역량이 필요해요.

코딩 + 자동화

QA 엔지니어

품질 프로세스와 테스트 전략을 설계하고 품질 지표를 관리해요. 릴리스 가부 판단의 기준을 세워요.

전략 + 프로세스

채용 절차와 코딩 테스트 구성

1. 서류

이력서, 포트폴리오, 테스트 경험

2. 코딩 테스트

알고리즘(쉬움~중간), SQL, 가끔 테스트 케이스 설계

3. 기술 면접

테스트 이론, CS 기초, 상황형 질문

4. 인성/임원

협업, 커뮤니케이션, 컬처핏

코딩 테스트 팁. QA 코테 알고리즘은 보통 개발자보다 쉬운 편이라 문자열, 배열, 해시, 정렬, 구현을 탄탄히 하면 충분한 경우가 많아요. 대신 SQL은 비중이 커서 따로 준비해 두면 유리해요.

역량 로드맵

단계준비할 것
기초테스트 이론(설계 기법, 레벨, 유형), 기본 알고리즘, SQL SELECT
자동화언어 1개(Python 또는 Java), Selenium 또는 Playwright, API 테스트(Postman)
심화CI/CD 안에서의 테스트, 성능/부하 테스트, 보안 테스트 기초, 테스트 전략 수립
이론

테스트 이론

면접에서 가장 많이 묻는 영역이에요. 용어를 내 말로 설명할 수 있어야 해요.

테스트 레벨

레벨대상주로 누가
단위 테스트함수, 클래스 같은 최소 단위개발자
통합 테스트모듈 간 인터페이스와 상호작용개발자, QA
시스템 테스트완성된 시스템 전체 동작QA
인수 테스트사용자 요구사항 충족과 인도 여부사용자, 발주자

테스트 유형

기능 테스트는 무엇을 하는지(요구한 기능이 동작하는지)를 봐요. 비기능 테스트는 얼마나 잘 하는지를 봐요.

성능 / 부하

예상 부하에서 응답 시간과 처리량을 측정해요.

스트레스

한계를 초과하는 부하로 한계점과 복구 동작을 확인해요.

보안

인증, 권한, 취약점을 점검해요.

사용성 / 호환성

사용 편의성, 그리고 여러 브라우저와 기기에서의 동작을 확인해요.

테스트 프로세스

1. 계획

범위, 일정, 자원

2. 분석/설계

테스트 케이스 작성

3. 구현/실행

테스트 수행, 결함 기록

4. 평가/리포트

결과 분석, 보고

5. 종료

산출물 정리, 회고

V모델. 요구사항 분석, 설계, 구현 같은 개발 단계마다 인수, 시스템, 단위 테스트가 짝을 이뤄요. 테스트를 개발 초기부터 계획한다는 점이 핵심이에요. 정적 테스트는 코드를 실행하지 않고 리뷰로, 동적 테스트는 실행해서 결함을 찾아요.

테스트 자동화

자동화 피라미드는 빠르고 안정적인 단위 테스트를 가장 많이, 느리고 깨지기 쉬운 E2E 테스트를 가장 적게 두기를 권장해요. 반복되는 회귀, 스모크 테스트가 자동화에 적합하고, UI 변동이 잦거나 일회성인 것은 수동이 더 효율적이에요.

분류대표 도구
UI 자동화Selenium, Playwright, Cypress
API 테스트Postman, REST Assured
단위 테스트JUnit(Java), pytest(Python)

테스트 설계 기법

적은 케이스로 많은 결함을 잡는 대표 기법이에요.

테스트 이론 퀴즈

실무

버그 리포트와 테스트 케이스 작성

QA의 실무 산출물이에요. 코테나 과제에서 직접 작성하라고 요구하기도 해요.

좋은 버그 리포트의 구성요소

항목내용
제목무엇이 어디서 잘못됐는지 한 줄로 요약
환경OS, 브라우저, 버전, 계정 등
사전 조건재현에 필요한 상태
재현 단계1, 2, 3 번호로 누구나 따라 할 수 있게
기대 결과정상이라면 어떻게 동작해야 하는지
실제 결과실제로 어떤 일이 일어났는지
심각도/우선순위영향 크기와 처리 순서
첨부스크린샷, 영상, 로그
심각도(Severity)와 우선순위(Priority)는 달라요. 심각도는 결함이 시스템에 미치는 영향의 크기, 우선순위는 얼마나 빨리 고쳐야 하는지예요. 자주 보는 회사 로고의 오타는 심각도는 낮지만 우선순위는 높을 수 있어요.

먼저 스스로 적어 본 뒤 아래 모범 예시와 비교해요. 정상 흐름만이 아니라 비정상과 경계까지 떠올리는 게 핵심이에요.

유형테스트 케이스 예시
정상올바른 아이디와 비밀번호로 로그인 성공
비정상틀린 비밀번호, 존재하지 않는 아이디, 빈 입력
경계아이디/비밀번호 최소, 최대 길이와 그 직전직후
보안연속 실패 시 계정 잠금, SQL 삽입 같은 악성 입력 처리
상태로그인 후 새로고침 시 세션 유지, 로그아웃 후 접근 차단

테스트 케이스의 구성요소

TC ID, 제목, 사전 조건, 입력과 절차, 기대 결과, 우선순위로 구성해요. 누가 실행해도 같은 결과가 나오도록 명확하게 적는 것이 좋은 테스트 케이스예요.
