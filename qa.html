직무

QA 직무와 채용

QA도 직무에 따라 코딩 테스트와 면접에서 보는 것이 달라요. 내 목표 직무를 먼저 정하고, 거기에 맞춰 준비 범위를 좁히는 게 효율적이에요.

QA 직무 세 갈래

수동 QA (Manual QA)

명세와 시나리오 기반으로 직접 테스트하고 탐색적 테스트로 결함을 찾아요. 버그 리포트와 테스트 케이스 작성이 핵심이에요.

테스트 이론 중심

자동화 QA / SDET

테스트를 코드로 작성해요. UI 자동화(Selenium, Playwright), API 테스트, 단위 테스트, CI 연동을 다뤄요. 프로그래밍 역량이 필요해요.

코딩 + 자동화

QA 엔지니어

품질 프로세스와 테스트 전략을 설계하고 품질 지표를 관리해요. 릴리스 가부 판단의 기준을 세워요.

전략 + 프로세스

채용 절차와 코딩 테스트 구성

1. 서류

이력서, 포트폴리오, 테스트 경험

2. 코딩 테스트

알고리즘(쉬움~중간), SQL, 가끔 TC 설계

3. 기술 면접

테스트 이론, CS 기초, 상황형 질문

4. 인성/임원

협업, 커뮤니케이션, 컬처핏

코딩 테스트 팁. QA 코테 알고리즘은 보통 개발자보다 쉬운 편이라 문자열, 배열, 해시, 정렬, 구현을 탄탄히 하면 충분한 경우가 많아요. 대신 SQL은 비중이 커서 따로 준비해 두면 유리해요.

역량 로드맵

단계준비할 것
기초테스트 이론(설계 기법, 레벨, 유형), 기본 알고리즘, SQL SELECT
자동화언어 1개(Python 또는 Java), Selenium 또는 Playwright, API 테스트(Postman)
심화CI/CD 안에서의 테스트, 성능/부하 테스트, 보안 테스트 기초, 테스트 전략 수립

한눈에 정리

  • 직무 : 수동 QA, 자동화 QA/SDET, QA 엔지니어로 나뉘고 준비 범위가 달라요.
  • 코테 : 알고리즘은 기본기 위주, SQL 비중이 커요.
  • 면접 : 테스트 이론과 CS 기초, 상황형 질문이 핵심이에요.
  • 먼저 할 일 : 목표 직무를 정하고 그에 맞는 깊이로 준비하세요.
테스트 이론

테스트 레벨과 유형

면접에서 가장 많이 묻는 영역이에요. 무엇을, 어느 단위로, 어떤 관점에서 테스트하는지 용어를 내 말로 설명할 수 있어야 해요.

테스트 레벨 (무엇을 테스트하나)

개발 단위가 커지는 순서로 네 단계가 있어요. 작은 단위부터 검증해 올라가면 결함을 더 일찍, 더 싸게 잡을 수 있어요.

레벨대상주로 누가
단위 테스트함수, 클래스 같은 최소 단위개발자
통합 테스트모듈 간 인터페이스와 상호작용개발자, QA
시스템 테스트완성된 시스템 전체 동작QA
인수 테스트사용자 요구사항 충족과 인도 여부사용자, 발주자

테스트 유형 (어떤 관점으로 보나)

기능 테스트는 무엇을 하는지(요구한 기능이 동작하는지)를 봐요. 비기능 테스트는 얼마나 잘 하는지를 봐요.

성능 / 부하

예상 부하에서 응답 시간과 처리량을 측정해요.

스트레스

한계를 초과하는 부하로 한계점과 복구 동작을 확인해요.

보안

인증, 권한, 취약점을 점검해요.

사용성 / 호환성

사용 편의성, 그리고 여러 브라우저와 기기에서의 동작을 확인해요.

정적 테스트와 동적 테스트, 그리고 V모델

정적 테스트는 코드를 실행하지 않고 리뷰나 정적 분석으로 결함을 찾고, 동적 테스트는 실제로 실행해 동작을 확인해요.

V모델. 요구사항 분석, 설계, 구현 같은 개발 단계마다 인수, 시스템, 단위 테스트가 좌우로 짝을 이뤄요. 테스트를 개발이 끝난 뒤가 아니라 초기부터 함께 계획한다는 점이 핵심이에요.

한눈에 정리

  • 레벨 : 단위 -> 통합 -> 시스템 -> 인수 순으로 범위가 커져요.
  • 유형 : 기능(무엇을)과 비기능(얼마나 잘)으로 나뉘어요.
  • 정적 vs 동적 : 실행하지 않는 리뷰가 정적, 실행해 보는 것이 동적이에요.
  • V모델 : 단계별로 테스트를 짝지어 초기부터 계획해요.
테스트 이론

테스트 프로세스와 자동화

테스트는 즉흥이 아니라 계획에서 종료까지 흐름을 따라요. 그리고 반복되는 부분은 자동화로 효율을 높여요.

테스트 프로세스

1. 계획

범위, 일정, 자원

2. 분석/설계

테스트 케이스 작성

3. 구현/실행

테스트 수행, 결함 기록

4. 평가/리포트

결과 분석, 보고

5. 종료

산출물 정리, 회고

테스트 자동화 피라미드

자동화 피라미드는 빠르고 안정적인 단위 테스트를 가장 많이, 느리고 깨지기 쉬운 E2E 테스트를 가장 적게 두기를 권장해요. 위로 갈수록 실행이 느리고 유지보수가 어려워지기 때문이에요.

분류대표 도구
UI 자동화 (E2E)Selenium, Playwright, Cypress
API 테스트Postman, REST Assured
단위 테스트JUnit(Java), pytest(Python)

무엇을 자동화하나

반복되는 회귀 테스트와 스모크 테스트, 입력과 기대 결과가 명확하고 안정적인 기능이 자동화에 적합해요. 반대로 UI 변동이 잦거나 일회성, 탐색적 테스트는 수동이 더 효율적이에요.

회귀 테스트. 코드 변경이나 버그 수정 후, 기존 기능이 그대로 동작하는지 다시 확인하는 테스트예요. 반복 실행이 많아 자동화의 1순위 대상이에요.

CI/CD 안에서의 테스트

코드를 올릴 때마다 빌드와 테스트를 자동으로 돌리면(지속적 통합) 결함을 빠르게 발견해요. 자동화된 테스트가 통과해야 다음 단계로 넘어가도록 품질 게이트를 두는 것이 좋아요.

한눈에 정리

  • 프로세스 : 계획 -> 분석/설계 -> 실행 -> 평가 -> 종료.
  • 피라미드 : 단위 많이, E2E 적게.
  • 자동화 대상 : 반복되는 회귀/스모크, 안정적인 기능.
  • 수동이 나은 것 : 탐색적 테스트, UI 변동이 잦거나 일회성.
테스트 이론

테스트 설계 기법

모든 입력을 다 시험할 수는 없어요. 적은 케이스로 많은 결함을 잡는 대표 기법들을 익혀, 빠짐 없이 효율적으로 케이스를 뽑아요.

동등 분할 (Equivalence Partitioning)

같은 결과를 내는 입력들을 하나의 그룹(클래스)으로 묶고, 그룹마다 대표값 하나만 시험해요. 그룹 안의 값은 동작이 같다고 보기 때문에 모두 시험할 필요가 없어요.

예시. 점수 입력이 0~100이라면 "유효(0~100)", "너무 작음(음수)", "너무 큼(101 이상)" 세 그룹으로 나눠 각 그룹 대표값 하나씩만 시험해요.

경계값 분석 (Boundary Value Analysis)

결함은 그룹의 경계 부근에서 자주 생겨요. 그래서 경계인 최소, 최대 값과 그 바로 직전직후 값을 집중 시험해요. 동등 분할과 짝을 이뤄 쓰면 효과가 커요.

예시. 허용 범위가 1~100이면 경계인 1과 100, 그리고 바로 바깥인 0과 101을 함께 시험해요.

결정 테이블 (Decision Table)

여러 조건의 조합에 따라 결과가 달라질 때, 조건과 결과를 표로 정리해 빠짐 없이 케이스를 만들어요. 조건이 많아 말로 헷갈릴 때 특히 유용해요.

회원 여부쿠폰 보유결과(할인)
회원있음둘 다 적용
회원없음회원 할인만
비회원있음쿠폰만
비회원없음할인 없음

상태 전이 (State Transition)

상태가 있는 기능에서 "어떤 상태에서 어떤 동작을 하면 어떤 상태로 바뀌는지"를 시험해요. 정상 전이뿐 아니라 허용되지 않는 전이(예: 로그아웃 상태에서 결제)도 확인해요.

예시. 주문 상태가 결제 대기 -> 결제 완료 -> 배송 중 -> 완료로 흐를 때, 건너뛰는 전이나 역방향 전이가 막히는지 시험해요.

페어와이즈 (Pairwise)

설정 항목이 많으면 모든 조합을 다 시험하기 어려워요. 페어와이즈는 "어떤 두 항목의 값 조합"이 적어도 한 번씩은 나오도록 케이스를 줄여, 적은 수로 대부분의 조합 결함을 잡아요.

한눈에 정리

  • 동등 분할 : 같은 결과 그룹은 대표값 하나만.
  • 경계값 분석 : 경계와 그 직전직후를 집중 시험(동등 분할과 짝).
  • 결정 테이블 : 조건 조합을 표로 빠짐 없이.
  • 상태 전이 : 허용/비허용 전이를 모두 확인.
  • 페어와이즈 : 두 항목 조합 기준으로 케이스를 줄여요.
실무

버그 리포트와 테스트 케이스 작성

QA의 실무 산출물이에요. 코테나 과제에서 직접 작성하라고 요구하기도 하고, 면접에서도 자주 물어봐요.

좋은 버그 리포트의 구성요소

항목내용
제목무엇이 어디서 잘못됐는지 한 줄로 요약
환경OS, 브라우저, 버전, 계정 등
사전 조건재현에 필요한 상태

심각도(Severity)와 우선순위(Priority)는 달라요. 심각도는 결함이 시스템에 미치는 영향의 크기, 우선순위는 얼마나 빨리 고쳐야 하는지예요. 메인 화면 로고의 오타는 심각도는 낮지만 우선순위는 높을 수 있어요.

좋은 테스트 케이스

TC ID, 제목, 사전 조건, 입력과 절차, 기대 결과, 우선순위로 구성해요. 누가 실행해도 같은 결과가 나오도록 명확하게, 요구사항에 추적 가능하게 적는 것이 좋은 테스트 케이스예요.

한눈에 정리

  • 버그 리포트 : 제목, 환경, 재현 단계, 기대/실제 결과, 심각도/우선순위가 빠지면 안 돼요.

CS 기초

CS 핵심

QA 면접에서도 CS 기초를 물어봐요. 자료구조, 데이터베이스, 네트워크, 운영체제의 핵심만 짚어 정리했어요. 깊이보다 "정확히 한 문장으로 설명하기"가 목표예요.

자료구조

구조핵심
배열인덱스로 임의 접근 O(1), 메모리 연속. 중간 삽입/삭제는 O(n).
연결 리스트노드를 포인터로 연결. 위치를 알면 삽입/삭제 O(1), 임의 접근은 O(n).
스택LIFO, 나중에 넣은 것이 먼저 나와요. 괄호 검사, 되돌리기.
큐FIFO, 먼저 넣은 것이 먼저 나와요. BFS, 작업 대기열.
해시 테이블키를 해시로 매핑해 평균 O(1) 탐색. 서로 다른 키가 같은 칸에 가면 충돌.
이진 탐색 트리왼쪽 < 부모 < 오른쪽. 균형이 깨지면 최악 O(n).
힙부모가 자식보다 작거나(최소 힙) 큰 완전 이진 트리. 루트가 최솟값/최댓값.
자주 나오는 함정. 연결 리스트는 임의 접근이 느리고(O(n)), 해시는 충돌을 체이닝이나 개방 주소법으로 해결해요. BST와 힙은 둘 다 트리지만 목적이 달라요(탐색 vs 최댓값/최솟값).

데이터베이스

개념핵심
인덱스정렬된 자료구조(보통 B-tree)로 검색을 빠르게. 대신 삽입/갱신 유지 비용이 있어요.
정규화중복을 줄여 삽입/갱신/삭제 이상 현상을 방지해요.
ACID원자성(전부 아니면 전무), 일관성, 격리성, 지속성.
JOININNER는 일치하는 행만, OUTER는 일치하지 않는 행도 포함.
기본키 / 외래키기본키는 행을 유일 식별(중복, NULL 불가). 외래키는 다른 테이블을 참조.
SQL vs NoSQL관계형은 스키마와 조인, 강한 트랜잭션에 강해요. NoSQL은 유연한 스키마와 수평 확장에 유리.

네트워크

개념핵심
TCP vs UDPTCP는 연결지향, 순서 보장과 재전송으로 신뢰성. UDP는 비연결형, 빠르지만 비신뢰.
3-way handshakeSYN -> SYN-ACK -> ACK 순으로 연결을 맺어요.
HTTP vs HTTPSHTTPS는 TLS(이전 SSL)로 암호화. 포트도 80과 443으로 달라요.
상태 코드200 성공, 404 자원 없음, 403 권한 없음, 500 서버 오류.
DNS도메인 이름을 IP 주소로 변환해요.
OSI 계층네트워크 계층이 IP 라우팅, 전송 계층이 종단 간 신뢰성을 담당.

운영체제

개념핵심
프로세스 vs 스레드같은 프로세스의 스레드는 코드/데이터/힙을 공유하고 스택은 따로. 그래서 전환 비용이 적어요.
교착 상태상호 배제, 점유와 대기, 비선점, 순환 대기 네 조건이 동시에 성립할 때 발생.
스케줄링FCFS는 비선점(도착 순), Round Robin과 SRT는 선점형.
뮤텍스 vs 세마포어뮤텍스는 한 번에 하나, 카운팅 세마포어는 정해진 수만큼 동시 접근 허용.
가상 메모리페이징으로 물리 메모리보다 큰 주소 공간을 제공. 주소 변환에 페이지 테이블.
컨텍스트 스위칭현재 프로세스 상태를 저장하고 다음 프로세스 상태를 복원. 오버헤드가 있어요.

한눈에 정리

  • DB : 인덱스로 검색 가속, 정규화로 중복 제거, ACID, JOIN 종류.
  • 네트워크 : TCP 신뢰/UDP 속도, 3-way handshake, HTTPS 암호화, 상태 코드.
  • OS : 프로세스와 스레드, 교착 상태 네 조건, 스케줄링, 동기화.
