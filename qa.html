QA 직무와 채용

QA도 직무에 따라 코딩 테스트와 면접에서 보는 것이 달라요. 내 목표 직무를 먼저 정하고, 거기에 맞춰 준비 범위를 좁히는 게 효율적이에요.

QA 직무 세 갈래

수동 QA (Manual QA)

명세와 시나리오 기반으로 직접 테스트하고 탐색적 테스트로 결함을 찾아요. 버그 리포트와 테스트 케이스 작성이 핵심이에요. 코딩은 필수가 아니에요.

테스트 이론 중심

자동화 QA / SDET

테스트를 코드로 작성해요. UI 자동화(Selenium, Playwright), API 테스트, CI 연동을 다뤄요. 사실상 개발 직군에 가까워 코딩 테스트가 있어요.

코딩 + 자동화

QA 엔지니어

품질 프로세스와 테스트 전략을 설계하고 품질 지표를 관리해요. 릴리스 가부 판단의 기준을 세워요.

전략 + 프로세스

채용 절차와 코딩 테스트 구성

1. 서류

이력서, 포트폴리오, 테스트 경험

2. 코딩 테스트

알고리즘(쉬움~중간), SQL, 가끔 TC 설계

3. 기술 면접

테스트 이론, CS 기초, 상황형 질문

4. 인성/임원

협업, 커뮤니케이션, 컬처핏

코딩 테스트 팁. QA 코테 알고리즘은 보통 개발자보다 쉬운 편이라 문자열, 배열, 해시, 정렬, 구현을 탄탄히 하면 충분한 경우가 많아요. 대신 SQL은 비중이 커서 따로 준비해 두면 유리해요.

한눈에 정리

  • 직무 : 수동 QA, 자동화 QA/SDET, QA 엔지니어로 나뉘고 준비 범위가 달라요.
  • 코테 : 알고리즘은 기본기 위주, SQL 비중이 커요.
  • 면접 : 테스트 이론과 CS 기초, 상황형 질문이 핵심이에요.
  • 먼저 할 일 : 목표 직무를 정하고 그에 맞는 깊이로 준비하세요. (16번 주제에서 공고 요구사항을 자세히 다뤄요.)
테스트 이론

소프트웨어 테스트의 기초

테스트는 소프트웨어가 의도대로 동작하는지 평가하고 검증하는 활동이에요. 목적은 단순히 "버그 찾기"가 아니라, 결함을 가능한 한 일찍 찾아 위험과 비용을 낮추고, 지금 품질이 어느 수준인지에 대한 객관적 정보로 릴리스 의사결정을 돕는 데 있어요. 면접에서 "테스트가 왜 필요하냐"를 물으면 이 관점으로 답해야 좋은 인상을 줘요.

테스트의 목적

  • 출시 전에 결함을 발견해 운영 장애와 사고 위험을 줄여요.
  • 요구사항 충족(검증)과 사용자/비즈니스 요구 적합성(확인)을 함께 확인해요.
  • 품질 수준에 대한 신뢰를 확보하고, 릴리스 가부 같은 의사결정에 근거를 제공해요.
  • 리뷰와 정적 분석으로 결함을 예방하고, 계약/법규/표준 준수를 입증해요.

오류 -> 결함 -> 실패 (용어의 인과 사슬)

ISTQB에서 가장 기본이 되는 용어 체인이에요. 셋을 정확히 구분해서 말할 수 있어야 해요.

용어정의예시
오류 (Error / Mistake)사람이 저지르는 실수. 잘못된 이해나 부주의.개발자가 경계 조건을 <= 대신 < 로 잘못 이해
결함 (Defect / Bug / Fault)그 실수가 코드나 문서에 남긴 잘못된 부분.소스에 if (age < 19) 로 잘못 작성됨
실패 (Failure)결함이 실행되어 실제로 드러난 잘못된 동작.만 19세 사용자가 성인 인증에 실패
핵심. 사람의 오류가 코드에 결함을 만들고, 그 결함이 실행될 때 실패가 관측돼요. 결함이 있어도 그 코드 경로가 실행되지 않으면 실패는 안 보일 수 있어요. 그래서 "통과 = 무결함"이 아니에요.

정적 테스트 vs 동적 테스트

코드를 실행하지 않고 문서/코드를 검토하는 게 정적 테스트, 실행해서 결과를 확인하는 게 동적 테스트예요. 정적 테스트는 결함을 더 이른 단계에서, 더 싸게 잡아요.

정적 리뷰 종류공식성특징
인포멀 리뷰낮음동료가 가볍게 훑어봄. 기록 없이 빠르게.
워크스루 (Walkthrough)중간작성자가 주도해 설명하며 검토. 학습/합의 목적.
기술 리뷰 (Technical Review)중간~높음동료 전문가들이 기술적 적합성을 검토.
인스펙션 (Inspection)가장 높음정해진 역할(모더레이터 등)과 절차로 지표까지 남기는 가장 엄격한 리뷰.

정적 분석 도구(SonarQube, ESLint 등)는 실행 없이 코드 규칙 위반/잠재 버그/복잡도를 자동으로 잡아 정적 테스트를 보완해요.

ISTQB 테스트 7원칙

원칙의미
1. 결함의 존재를 보여줄 뿐테스트는 결함이 있음을 보여줄 수 있어도, 결함이 없음을 증명하지는 못해요. 통과가 무결함의 증거는 아니에요.
2. 전수 테스트는 불가능모든 입력과 조건 조합을 다 시험할 수는 없어요. 위험 분석과 우선순위로 대체해요.
3. 조기 테스트 (시프트 레프트)생애주기 초반부터 테스트하면 결함을 더 싸고 빠르게 잡아요.
4. 결함 집중 (군집)소수 모듈에 대부분의 결함이 몰려요(파레토 80/20). 그 모듈에 테스트를 더 투입해요.
5. 살충제 역설같은 테스트만 반복하면 새 결함을 못 찾아요. 테스트를 주기적으로 갱신/추가해야 해요.
6. 정황(맥락) 의존안전이 중요한 시스템(의료/항공)과 일반 앱은 테스트 강도와 방식이 달라요.
7. 오류 부재는 착각결함을 다 잡아도 사용자 요구를 못 채우면 의미가 없어요(요구 자체가 틀렸을 수도).
자주 헷갈리는 4번 vs 5번. 결함 집중은 "결함이 특정 모듈에 몰린다"는 관찰이고, 살충제 역설은 "같은 케이스를 반복하면 효과가 떨어진다"는 현상이에요. 둘을 바꿔 답하지 않게 조심해요.

조기 테스트가 왜 중요한가 (결함 비용 곡선)

결함은 늦게 발견될수록 수정 비용이 기하급수적으로 커져요. 요구 단계에서 잡으면 문장 한 줄 고치면 되지만, 운영 중 사고로 터지면 데이터 복구/장애 대응/신뢰 하락까지 얹혀요.

발견 시점상대적 수정 비용(대략)
요구/설계1x
개발/코딩수 배
테스트(QA)10배 안팎
운영(출시 후)수십~수백 배
실무 연결. 그래서 요구/설계 리뷰(정적 테스트)와 시프트 레프트가 강조돼요. "테스트는 마지막에 하는 것"이 아니라 처음부터 함께 가는 활동이에요.

헷갈리기 쉬운 용어

구분핵심
검증(Verification) vs 확인(Validation)검증 = "제품을 올바르게 만들고 있나?"(명세 부합, 주로 정적 리뷰). 확인 = "올바른 제품을 만들고 있나?"(사용자 요구 충족, 주로 동적 실행). 스펙대로 완벽히 만들어도(검증 통과) 스펙 자체가 틀렸으면 확인에서 걸려요.
QA vs QC vs 테스팅QA는 프로세스 지향 예방 활동, QC는 산출물에서 결함을 찾는 검출 활동, 테스팅은 QC의 실행 수단 중 하나예요. 포함 관계는 QA가 QC를, QC가 테스팅을 품어요.

한눈에 정리

  • 테스트 : 결함을 일찍 찾아 위험/비용을 줄이고 릴리스 판단 근거를 주는 활동이에요.
  • 용어 사슬 : 오류(사람) -> 결함(코드) -> 실패(실행 시 관측).
  • 정적/동적 : 실행 안 함(리뷰/정적분석) vs 실행함. 정적이 더 싸고 이르게 잡아요.
  • 7원칙 : 전수 불가, 조기 테스트, 결함 집중, 살충제 역설이 단골이에요.
  • 검증 vs 확인 : 명세대로 만드는가 vs 올바른 것을 만드는가. QA ⊃ QC ⊃ 테스팅.
테스트 이론

개발 생명주기 (SDLC)

소프트웨어는 요구 분석부터 유지보수까지 일정한 흐름으로 만들어져요. 어떤 모델을 쓰느냐에 따라 QA가 언제, 어떻게 참여하는지가 달라져요. 면접에서 "워터폴과 애자일의 차이", "V모델이 뭐냐"는 단골 질문이에요.

SDLC vs STLC. SDLC는 개발 전체의 생명주기(요구-설계-구현-테스트-배포-유지보수)이고, STLC는 그 안에서 테스트 활동만 떼어낸 흐름이에요. SDLC 안에 STLC가 들어 있다고 보면 돼요.

대표 모델과 QA 관여 시점

모델특징QA 관여
워터폴요구 -> 설계 -> 구현 -> 테스트 -> 배포 순서. 단계가 끝나야 다음으로. 요구가 명확하고 변경이 적은 프로젝트에 적합. 변경에 약하고 문서 중심.늦음. 개발이 끝난 뒤 테스트 단계가 시작돼 결함을 늦고 비싸게 발견.
V모델워터폴을 V자로 펼친 형태. 좌측 개발 단계마다 우측에 대응 테스트가 짝을 이뤄요. 검증/확인 중심.이르고 병렬. 각 개발 단계와 나란히 테스트를 계획/설계(실행은 대응 단계에서).
반복/점진짧은 반복으로 기능을 조금씩 쌓아 완성. 변경을 수용.매 반복 종료 시. 회귀 테스트로 기존 기능을 보호.
스파이럴반복마다 리스크 분석을 먼저 하는 리스크 주도 모델. 크고 위험이 큰 프로젝트에 적합.반복마다 리스크 기반으로 테스트 우선순위를 정해요.
애자일2~4주 스프린트로 지속 인도. 변경을 환영하고 협업 중심.매 스프린트 내내 지속적. 개발과 나란히 테스트하고 자동화/CI와 결합.
데브옵스개발과 운영을 통합해 CI/CD로 자주 배포. 애자일의 확장.파이프라인에 자동 테스트를 심어 배포 게이트로 삼아요(시프트 레프트+라이트).
V모델의 짝. 요구 분석은 인수 테스트와, 시스템 설계는 시스템 테스트와, 아키텍처/통합 설계는 통합 테스트와, 상세(모듈) 설계는 단위 테스트와 짝을 이뤄요. 왼쪽에서 문서를 만들 때 오른쪽 테스트를 미리 설계한다는 점이 핵심이에요.

어떤 모델을 언제 쓰나

  • 워터폴/V모델 : 요구가 확정적이고 규제/문서가 중요한 경우(금융, 국방, 의료).
  • 애자일/데브옵스 : 요구가 자주 바뀌고 빠른 피드백이 필요한 경우(웹/앱 서비스). 요즘 주류.
  • 스파이럴 : 실패 비용이 크고 불확실성이 높아 리스크를 단계적으로 줄여야 할 때.

한눈에 정리

  • 워터폴 : 순차적, QA가 가장 늦게 참여(결함을 늦고 비싸게).
  • V모델 : 개발 단계마다 테스트가 짝(단위/통합/시스템/인수)을 이뤄 일찍 설계해요.
  • 반복/점진/스파이럴 : 주기마다 회귀로 보호, 스파이럴은 리스크 주도.
  • 애자일/데브옵스 : 스프린트 내내 지속 테스트 + CI/CD 자동화(주류).
테스트 이론

애자일과 스크럼

요즘 대부분의 팀은 애자일로 일해요. 짧은 주기로 만들고 피드백받아 고쳐 나가요. QA는 마지막 관문이 아니라 처음부터 함께하는 품질 옹호자(quality advocate)예요. 스크럼 용어와 QA의 자리매김을 정확히 말할 수 있어야 해요.

애자일 선언의 4가지 가치

2001년 애자일 선언은 오른쪽도 가치 있지만 왼쪽을 더 중시한다고 밝혀요.

  • 공정과 도구보다 개인과 상호작용
  • 포괄적 문서보다 작동하는 소프트웨어
  • 계약 협상보다 고객과의 협력
  • 계획을 따르기보다 변화에 대응

스크럼의 3역할

스프린트는 사용 가능한 결과물(증분)을 만드는 고정 길이(타임박스) 주기예요. 보통 1~4주이고 2주가 가장 흔해요. 길이는 일정하게 유지해요.

역할책임
제품 책임자(PO)제품 가치 극대화. 제품 백로그를 소유하고 우선순위를 정해요. "무엇을, 왜".
스크럼 마스터스크럼이 잘 돌아가게 코칭하고 장애물을 제거해요. "어떻게 일할지".
개발팀자기조직적으로 스프린트 작업을 계획하고 증분을 만들어요. "만들기".

3가지 산출물(아티팩트)과 완료의 정의

아티팩트의미
제품 백로그제품에 필요한 모든 일의 우선순위 목록. PO가 소유하고 계속 다듬어요.
스프린트 백로그이번 스프린트에 뽑아온 항목 + 그것을 완성할 계획.
증분(Increment)이번 스프린트에서 완성해 잠재적으로 출시 가능한 결과물.
DoR vs DoD. 준비의 정의(DoR)는 "이 항목을 스프린트에 넣어도 될 만큼 명확한가"의 기준이고, 완료의 정의(DoD)는 "이 항목을 끝났다고 부를 수 있는가"의 기준이에요. QA 관점에서 DoD에는 보통 "인수 기준 충족 + 테스트 통과 + 회귀 이상 없음"이 들어가요.

주요 이벤트(세리머니)

스프린트 계획

목표 합의, 할 일 선정

데일리 스탠드업

15분 타임박스, 진척과 블로커 점검

스프린트 리뷰

증분 시연, 이해관계자 피드백

회고

사람/프로세스/도구 개선

칸반과의 차이. 스크럼은 고정 스프린트로 끊어 일하고, 칸반은 스프린트 없이 흐름(WIP 제한)으로 연속 처리해요. 지속적 유지보수/운영에는 칸반이 잘 맞아요.

애자일 테스팅 4분면

Brian Marick의 4분면은 "무엇을 위해, 무엇을 테스트하는지"를 정리해요. 면접에서 애자일 테스트 전략을 물으면 유용해요.

분면목적예시
Q1 (팀 지원 / 기술)개발을 뒷받침단위/컴포넌트 테스트, TDD
Q2 (팀 지원 / 비즈니스)요구를 명확히기능 테스트, 인수 테스트, BDD 예시
Q3 (제품 비평 / 비즈니스)사용자 관점 검증탐색적 테스트, 사용성/UAT
Q4 (제품 비평 / 기술)비기능 품질성능/부하/보안 테스트, 도구 활용

QA의 역할: 시프트 레프트와 시프트 라이트

품질은 특정 부서가 아니라 팀 전체의 책임이에요. 테스트는 개발이 끝난 뒤 따로 하는 단계가 아니라 매 스프린트 내내 함께 진행돼요. QA는 자동화를 만들고, 리스크를 자문하고, 인수 기준을 명확히 하는 능동적 역할을 해요.

방향의미
시프트 레프트테스트를 초반(요구/설계/코딩)으로 당겨 결함을 예방. 요구 리뷰, 인수 기준 정의, 단위 자동화.
시프트 라이트출시 이후 운영에서도 품질을 관측. 모니터링/로그/알림, 카나리/A-B, 실사용 피드백.

한눈에 정리

  • 4가치 : 상호작용 / 작동 SW / 고객 협력 / 변화 대응을 더 중시해요.
  • 3역할 : PO(무엇을), 스크럼 마스터(어떻게), 개발팀(만들기).
  • 3아티팩트 : 제품 백로그 / 스프린트 백로그 / 증분. DoD에 테스트 통과 포함.
  • 이벤트 : 계획 / 데일리 / 리뷰 / 회고 (모두 타임박스).
  • QA : 팀 전체가 품질 책임. 시프트 레프트(예방) + 시프트 라이트(운영 관측).
테스트 이론

테스팅 프로세스 (STLC)

테스트도 즉흥이 아니라 정해진 흐름을 따라요. 요구사항을 검토하는 것부터 결과를 보고하고 마무리하는 것까지, 단계마다 할 일과 산출물이 있어요.

STLC 6단계

단계하는 일산출물
1. 요구사항 분석테스트 관점에서 요구사항을 검토하고 테스트 가능 여부를 파악, 모호한 점을 명확히.요구사항 추적표(RTM)
2. 테스트 계획전략, 범위, 일정, 자원, 도구, 공수를 정해요. 가장 중요한 단계예요.테스트 계획서
3. 케이스 설계입력, 절차, 기대 결과가 담긴 케이스를 작성하고 데이터를 준비, 동료 리뷰.테스트 케이스, 데이터
4. 환경 구축하드웨어, 소프트웨어, 데이터를 구성하고 스모크 테스트로 준비 확인.준비된 환경
5. 테스트 실행케이스를 수행하고 실제와 기대를 비교, 결함을 기록하고 재테스트.결과 로그, 결함 리포트
6. 종료결함 종결 확인, 지표 평가, 환경 정리, 회고.테스트 요약 보고서
RTM(요구사항 추적표). 어떤 요구사항이 어떤 테스트 케이스로 검증되는지 연결한 표예요. 빠진 요구사항이 없는지, 테스트가 요구를 모두 덮는지 확인하는 데 써요.

한눈에 정리

  • 흐름 : 요구 분석 -> 계획 -> 설계 -> 환경 -> 실행 -> 종료.
  • 가장 중요 : 테스트 계획 단계예요.
  • 추적 : RTM으로 요구와 테스트를 연결해 빠짐을 막아요.
  • SDLC와 다름 : SDLC는 개발 전체, STLC는 그 안의 테스트 흐름이에요.
테스트 이론

테스트 레벨과 유형

무엇을, 어느 단위로, 어떤 관점에서 테스트하는지 용어를 내 말로 설명할 수 있어야 해요. 면접에서 가장 많이 묻는 영역이에요.

테스트 레벨 (어느 단위를 보나)

레벨대상주로 누가
단위 테스트함수, 클래스 같은 최소 단위(스텁/목 활용)개발자
통합 테스트모듈 간 인터페이스와 상호작용개발자, QA
시스템 테스트완성된 전체 시스템을 처음부터 끝까지독립 테스트팀
인수 테스트사용자 요구와 인수 기준 대비 공식 검증사용자, 발주자

인수 테스트에는 현업이 검증하는 UAT, 사내에서 하는 알파, 외부 실사용자가 하는 베타 등이 있어요.

테스트 유형 (어떤 품질을 보나)

기능 테스트는 무엇을 하는지(요구한 기능이 동작하는지)를, 비기능 테스트는 얼마나 잘 하는지를 봐요.

유형핵심
성능여러 조건에서 속도, 응답성, 안정성, 자원 사용을 평가하는 우산 개념.
부하예상되는 정상에서 피크 범위의 부하에서 제대로 동작하는지.
스트레스정상/최대 용량을 초과한 부하로 한계점과 복구 동작을 확인.
보안인증, 권한, 취약점을 점검.
사용성 / 호환성사용 편의성, 그리고 여러 브라우저/기기에서의 일관 동작.
자주 나오는 구분. 성능은 전체를 아우르는 부모 개념이고, 부하는 예상 범위 안에서, 스트레스는 한계를 넘겨 임계점과 복구를 확인해요.

한눈에 정리

  • 레벨 : 단위 -> 통합 -> 시스템 -> 인수 순으로 범위가 커져요.
  • 유형 : 기능(무엇을)과 비기능(얼마나 잘)으로 나뉘어요.
  • 성능/부하/스트레스 : 전반 / 예상 범위 / 한계 초과로 구분해요.
테스트 이론

테스트 설계 기법

모든 입력을 다 시험할 수는 없어요. 적은 케이스로 많은 결함을 잡는 대표 기법을 익혀 빠짐 없이 효율적으로 케이스를 뽑아요. 명세를 보는 블랙박스와 코드를 보는 화이트박스로 나뉘어요.

블랙박스 (명세 기반)

동등 분할 (Equivalence Partitioning)

같은 결과를 내는 입력들을 한 그룹으로 묶고 그룹마다 대표값 하나만 시험해요. 그룹 안 값은 동작이 같다고 보기 때문이에요.

예시. 점수가 0~100이라면 "유효(0~100)", "너무 작음(음수)", "너무 큼(101 이상)" 세 그룹으로 나눠 대표값 하나씩 시험해요.

경계값 분석 (Boundary Value Analysis)

결함은 경계 부근에서 자주 생겨요. 그래서 경계인 최소, 최대와 그 바로 직전직후 값을 집중 시험해요. 동등 분할과 짝을 이루면 효과가 커요.

예시. 허용 범위가 1~100이면 1과 100, 그리고 바로 바깥인 0과 101을 함께 시험해요.

결정 테이블 / 상태 전이 / 페어와이즈

기법언제 / 어떻게
결정 테이블여러 조건의 조합에 따라 결과가 달라질 때, 조건과 결과를 표로 정리해 빠짐 없이.
상태 전이상태가 있는 기능에서 어떤 상태에서 어떤 동작을 하면 어떤 상태로 바뀌는지(허용/비허용 전이 모두).
페어와이즈설정 항목이 많을 때, 두 항목의 값 조합이 적어도 한 번씩 나오게 케이스를 줄여요.
유스케이스사용자 시나리오에서 기본 흐름과 대안/예외 흐름을 도출해요.

화이트박스 (구조 기반)

커버리지의미
구문(Statement)실행 가능한 코드 문장이 한 번 이상 실행된 비율. 셋 중 가장 약해요.
분기/결정(Branch)각 분기점을 참과 거짓 모두 실행한 비율. 구문보다 강해요.
조건(Condition)복합 조건 안의 개별 조건이 각각 참과 거짓으로 평가됐는지.
시험 단골 함정. 100% 분기 커버리지는 100% 구문 커버리지를 보장해요. 하지만 100% 조건 커버리지가 100% 결정(분기) 커버리지를 보장하지는 않아요.

한눈에 정리

  • 블랙박스 : 동등 분할(대표값), 경계값(경계 집중), 결정 테이블, 상태 전이, 페어와이즈.
  • 화이트박스 : 구문 < 분기 < 조건 순으로 촘촘해져요.
  • 핵심 : 동등 분할 + 경계값을 짝으로 쓰는 게 기본기예요.
테스트 이론

테스트 관리

무엇을 어떻게 테스트할지 계획하고, 얼마나 했는지 지표로 보고, 위험이 큰 곳에 자원을 집중하는 활동이에요. QA 엔지니어와 면접에서 자주 다뤄요.

테스트 계획과 전략

구분핵심
테스트 전략조직이나 프로그램 차원의 상위 접근(레벨, 방법론, 도구, 표준, 역할). "어떻게 테스트하는가"의 일반 원칙.
테스트 계획프로젝트나 릴리스별 문서(목표, 범위, 자원, 일정, 담당, 조건). "이번에 무엇을 테스트하는가". 전략에서 파생돼요.

테스트 추정

일정을 가늠하는 방법이에요. 작업을 잘게 나눠 합산하는 WBS, 전문가가 따로 추정 후 수렴하는 델파이, 그리고 3점 추정이 흔해요.

3점 추정. (낙관 + 4 x 최빈 + 비관) / 6 으로 계산해요. 최선과 최악, 가장 그럴듯한 값을 모두 반영해 한쪽으로 치우치지 않게 해요.

품질과 진척 지표

지표공식 / 의미
결함 밀도결함 수 / 규모(보통 천 줄). 낮을수록 좋아요.
테스트 커버리지테스트한 항목 / 전체 항목 x 100. 항목은 요구, 코드, 분기 등.
결함 제거 효율(DRE)테스트로 찾은 결함 / (테스트 발견 + 출시 후 발견) x 100. 100에 가까울수록 좋아요.

리스크 기반 테스트와 진입/종료 기준

위험은 발생 가능성과 영향의 곱이에요. 위험이 큰 영역을 먼저, 더 집중해서 테스트해요.

기준예
진입(Entry)환경 준비, 케이스/데이터 완비, 빌드 배포와 스모크 통과.
종료(Exit)계획한 케이스 실행 완료, 목표 커버리지 달성, 미해결 치명/높음 결함 없음, 일정/예산 충족.

한눈에 정리

  • 전략 vs 계획 : 일반 원칙(어떻게) vs 이번 프로젝트(무엇을).
  • 추정 : WBS, 델파이, 3점 추정(낙관/최빈/비관).
  • 지표 : 결함 밀도, 커버리지, DRE.
  • 리스크 기반 : 가능성 x 영향이 큰 곳을 먼저 테스트, 진입/종료 기준으로 관리.
기술 이해

앱 구조와 API

무엇을 테스트하는지 알려면 소프트웨어가 어떻게 동작하는지 먼저 알아야 해요. 화면, 서버, 데이터가 어떻게 주고받는지, API가 무엇인지 짚어요.

클라이언트-서버와 3계층 구조

클라이언트(브라우저나 앱)가 서버에 요청을 보내면, 서버가 처리해 응답을 돌려줘요. 보통 세 계층으로 나뉘어요.

계층역할
프론트엔드(화면)사용자가 보고 조작하는 화면(HTML/CSS/JS, React 등).
백엔드(로직)비즈니스 로직을 처리하고 API를 제공.
데이터베이스데이터를 영구 저장(MySQL 등 관계형, MongoDB 등 NoSQL).

흐름은 프론트 -> 백엔드 API 호출 -> DB 조회/저장 -> 응답 반환이에요.

API와 REST

API는 소프트웨어끼리 통신하는 규약이고, REST는 HTTP 기반으로 자원을 URL로 표현하는 방식이에요. 클라이언트가 요청하면 서버가 응답하는 구조예요.

HTTP 메서드역할(멱등 여부)
GET조회 (안전, 멱등)
POST생성 (멱등 아님)
PUT전체 수정/교체 (멱등)
PATCH부분 수정 (보통 멱등 아님)
DELETE삭제 (멱등)

멱등은 여러 번 호출해도 결과가 같다는 뜻이에요.

상태 코드의미
2xx성공 (200 OK, 201 Created, 204 No Content)
3xx리다이렉션 (301, 302)
4xx클라이언트 오류 (400 잘못된 요청, 401 인증 필요, 403 권한 없음, 404 없음)
5xx서버 오류 (500 내부 오류, 502, 503)

모바일 앱의 종류

종류특징
네이티브OS 전용 언어로 개발(iOS Swift, Android Kotlin). 성능과 기기 기능 접근이 최고, 플랫폼별 개발 필요.
하이브리드웹 기술로 만들어 네이티브 컨테이너로 감싸요. 한 코드로 양쪽, 성능은 네이티브보다 낮음.
웹앱브라우저에서 실행, 설치 불필요. 기기 기능 접근에 제한이 있어요.

한눈에 정리

  • 구조 : 클라이언트-서버, 프론트/백엔드/DB 3계층.
  • API/REST : 자원을 URL로, GET/POST/PUT/PATCH/DELETE로 다뤄요.
  • 상태 코드 : 2xx 성공, 4xx 클라이언트 오류, 5xx 서버 오류.
  • 모바일 : 네이티브 / 하이브리드 / 웹앱.
기술 이해

웹 테스트

웹 서비스는 다양한 브라우저, 화면 크기, 네트워크에서 똑같이 동작해야 해요. 무엇을 어디서 점검하는지 핵심 포인트를 정리했어요.

크로스 브라우징과 반응형

영역점검 포인트
크로스 브라우징Chrome, Firefox, Safari, Edge에서 렌더링(CSS), 동작(JS), 폰트/레이아웃 깨짐, 미지원 기능.
반응형화면 너비별 레이아웃 전환, 가로 스크롤 발생, 요소 겹침/잘림, 터치 타깃 크기, 가로/세로 회전.

쿠키, 세션, 폼 검증

영역점검 포인트
쿠키/세션로그인 유지와 만료, 타임아웃, 로그아웃 후 뒤로가기로 접근 차단, 동시 로그인.
캐시배포 직후 최신 리소스 반영, 이전 캐시로 인한 오작동.
폼/입력빈 값, 공백, 최대 길이 초과, 특수문자, 경계값, 필수 누락 시 에러, 중복 제출(더블 클릭).
클라이언트 검증은 우회될 수 있어요. 화면에서 막아도 서버가 다시 검증하는지 확인해야 해요. 개발자 도구로 클라이언트 검증을 우회한 뒤 서버가 거르는지 시험해 보세요.

웹 보안 기초 (테스터 관점)

취약점테스터가 확인할 것
XSS입력 필드나 URL에 스크립트를 넣어 실행되는지(댓글, 검색어, 프로필 등 입력이 화면에 출력되는 지점). 방어는 입력 escape.
SQL Injection로그인/검색창에 ' OR '1'='1 같은 값으로 인증 우회나 데이터 노출이 되는지. 방어는 매개변수화 쿼리.
CSRF정상 요청에 CSRF 토큰이 포함되는지, 토큰 없이/위조 시 거부되는지. 방어는 CSRF 토큰.

보조 도구로 OWASP ZAP(자동 스캔), Burp Suite(수동 가로채기)가 있고, 기준은 OWASP Top 10이에요.

성능과 접근성

영역점검 포인트
성능페이지 로드 시간, 첫 바이트까지 시간(TTFB), 이미지 최적화, 불필요한 요청 수, 느린 네트워크에서의 동작.
접근성(a11y)키보드만으로 사용 가능, 이미지 대체 텍스트, 폼 라벨 연결, 색 대비. 기준은 WCAG, 도구는 axe와 Lighthouse.

한눈에 정리

  • 호환 : 브라우저, 화면 크기, 회전까지 일관 동작.
  • 상태 : 쿠키/세션/캐시, 폼 입력의 정상과 비정상.
  • 보안 : XSS, SQL Injection, CSRF를 테스터 관점에서.
  • 품질 : 성능(로딩)과 접근성(키보드, 대체 텍스트)도 챙겨요.
기술 이해

CS 핵심

QA 면접에서도 CS 기초를 물어봐요. 자료구조, 데이터베이스, 네트워크, 운영체제의 핵심만 짚어 정리했어요. 깊이보다 정확히 한 문장으로 설명하기가 목표예요.

자료구조

구조핵심
배열 / 연결 리스트배열은 임의 접근 O(1), 중간 삽입 O(n). 연결 리스트는 그 반대.
스택 / 큐스택은 LIFO(나중에 넣은 게 먼저), 큐는 FIFO(먼저 넣은 게 먼저).
해시 테이블키를 해시로 매핑해 평균 O(1) 탐색. 다른 키가 같은 칸에 가면 충돌.
BST / 힙BST는 탐색용(왼쪽 < 부모 < 오른쪽), 힙은 최솟값/최댓값용(루트가 극값).

데이터베이스

개념핵심
인덱스정렬된 자료구조로 검색을 빠르게. 대신 삽입/갱신 유지 비용.
정규화중복을 줄여 삽입/갱신/삭제 이상을 방지.
ACID원자성(전부 아니면 전무), 일관성, 격리성, 지속성.
JOIN / 키INNER는 일치 행만. 기본키는 유일 식별(중복/NULL 불가), 외래키는 다른 테이블 참조.

네트워크

개념핵심
TCP vs UDPTCP는 연결지향, 순서/재전송으로 신뢰성. UDP는 빠르지만 비신뢰.
3-way handshakeSYN -> SYN-ACK -> ACK 로 연결을 맺어요.
HTTP vs HTTPSHTTPS는 TLS로 암호화. 포트도 80과 443으로 달라요.
DNS도메인 이름을 IP 주소로 변환.

운영체제

개념핵심
프로세스 vs 스레드같은 프로세스의 스레드는 코드/데이터/힙 공유, 스택은 따로. 전환 비용이 적어요.
교착 상태상호 배제, 점유와 대기, 비선점, 순환 대기 네 조건이 동시에 성립할 때.
뮤텍스 vs 세마포어뮤텍스는 한 번에 하나, 카운팅 세마포어는 정해진 수만큼 동시 접근.

한눈에 정리

  • 자료구조 : 스택 LIFO, 큐 FIFO, 해시 평균 O(1), BST와 힙의 목적 차이.
  • DB : 인덱스, 정규화, ACID, JOIN.
  • 네트워크 : TCP 신뢰/UDP 속도, 3-way handshake, HTTPS 암호화.
  • OS : 프로세스와 스레드, 교착 상태 네 조건, 동기화.

결함 생명주기

결함은 등록부터 종료까지 정해진 상태를 거쳐요.

New -> Assigned(배정) -> Open(수정 착수) -> Fixed(수정 완료) -> Retest(재검증) -> Verified -> Closed(종료)

분기 상태의미
Reopen수정했는데 다시 재현되면 열어서 개발자에게 되돌려요.
Deferred실제 결함이지만 급하지 않아 다음 릴리스로 연기.
Rejected재현 불가, 사양대로 동작, 중복 등으로 결함이 아니라고 판단.

심각도 vs 우선순위

둘은 달라요. 심각도는 시스템에 미치는 기술적 영향(보통 QA가 지정), 우선순위는 얼마나 빨리 고쳐야 하는지(PM/개발리드가 협의). 메인 화면 로고 오타는 심각도는 낮아도 우선순위는 높을 수 있어요.
조합예시
높은 심각도 + 높은 우선순위운영 환경 로그인 장애(전 사용자 접근 불가)
높은 심각도 + 낮은 우선순위거의 안 쓰는 환경에서만 나는 크래시
낮은 심각도 + 높은 우선순위로그인 화면 회사명 오타(모든 사용자가 봄)

자동화와 배포

테스트 자동화와 도구

반복되는 테스트는 사람이 매번 하기 어려워요. 자동화로 빠르고 일관되게 돌리되, 무엇을 자동화할지 고르는 눈이 중요해요.

자동화 피라미드와 무엇을 자동화하나

피라미드는 빠르고 안정적인 단위 테스트를 가장 많이, 느리고 깨지기 쉬운 E2E 테스트를 가장 적게 두기를 권장해요. 반복되는 회귀와 스모크, 입력과 기대 결과가 명확한 기능이 자동화에 적합해요. 반대로 UI 변동이 잦거나 일회성, 탐색적 테스트는 수동이 나아요.

UI 자동화 도구 비교

도구언어특징
SeleniumJava, Python, C#, JS 등 가장 넓음가장 오래된 표준, 생태계가 큼. 자동 대기가 약해 불안정(flaky) 위험, 설정이 복잡.
PlaywrightJS/TS, Python, .NET, Java자동 대기로 안정적이고 빠름. 크로스 브라우저, 병렬과 트레이싱 내장. 상대적으로 신생.
CypressJS/TS 전용개발자 친화, 실시간 디버깅. 단일 언어, 멀티탭/다중 브라우저에 한계.

API 테스트 도구

도구특징
PostmanGUI 기반, 코딩 없이 요청 작성과 테스트, 문서화. 진입장벽이 낮고 협업에 강함. CI 연동은 Newman.
REST AssuredJava 라이브러리. 코드로 작성하고 BDD 스타일(given-when-then). CI/CD 통합에 유리.

한눈에 정리

  • 피라미드 : 단위 많이, E2E 적게.
  • 자동화 대상 : 반복 회귀/스모크, 안정적인 기능.
  • UI 도구 : Selenium(넓음), Playwright(안정/빠름), Cypress(JS 전용).
  • API 도구 : Postman(GUI), REST Assured(코드).
자동화와 배포

Python으로 QA 자동화

QA 자동화에서 Python을 많이 써요. 문법이 쉽고 pytest, requests, Selenium 같은 도구가 잘 갖춰져 있어 API와 UI 테스트를 빠르게 만들 수 있어요.

pytest 기본 구조

파일명은 test_로 시작하고 함수명도 test_로 시작하면 자동으로 모여 실행돼요. 검증은 assert로 해요. 준비와 정리는 fixture로 분리해요.

import pytest

@pytest.fixture
def sample_user():
    return {"name": "kim", "age": 20}

def test_user_name(sample_user):   # fixture 를 인자로 요청
    assert sample_user["name"] == "kim"

def test_addition():
    assert 1 + 1 == 2

터미널에서 pytest (또는 pytest -v)로 실행해요.

requests 로 API 테스트

import requests

def test_get_user():
    res = requests.get("https://api.example.com/users/1")
    assert res.status_code == 200
    body = res.json()
    assert body["id"] == 1

def test_create_user():
    res = requests.post("https://api.example.com/users",
                        json={"name": "kim"})
    assert res.status_code == 201

res.status_code(상태 코드), res.json()(JSON 파싱), res.headers 등을 확인해요.

Selenium 으로 UI 자동화

from selenium import webdriver
from selenium.webdriver.common.by import By

driver = webdriver.Chrome()
driver.get("https://www.example.com")

box = driver.find_element(By.NAME, "q")   # 요소 찾기
box.send_keys("test")                      # 입력
driver.find_element(By.ID, "submit").click()
assert "결과" in driver.title
driver.quit()                              # 브라우저 종료
안정성 팁. 요소가 늦게 나타나면 테스트가 깨질 수 있어요. 실무에서는 WebDriverWait로 명시적 대기를 넣어 요소가 준비될 때까지 기다린 뒤 조작해요.

한눈에 정리

  • pytest : test_ 규칙, assert로 검증, fixture로 준비/정리.
  • requests : status_code와 json()으로 API 검증.
  • Selenium : By로 요소 찾기, 입력/클릭, 명시적 대기로 안정화.
자동화와 배포

CI/CD와 Jenkins

코드를 올릴 때마다 자동으로 빌드하고 테스트하면 문제를 빨리 발견해요. 자동화된 테스트가 파이프라인 안에서 품질의 문지기 역할을 해요.

CI와 CD

개념의미
CI (지속적 통합)코드를 자주 합치고, 합칠 때마다 자동으로 빌드와 테스트를 실행해 통합 오류를 일찍 발견.
CD (지속적 배포/전달)테스트를 통과한 코드를 자동으로 릴리스 준비하거나 운영까지 배포.

파이프라인에서 테스트의 위치

단계실행하는 테스트
커밋빠른 단위 테스트와 스모크(개발 흐름을 막지 않게 짧게)
빌드통합 테스트와 UI 회귀
릴리스전체 회귀, E2E, 성능, 보안
품질 게이트. 다음 단계로 넘어가기 전 충족해야 하는 조건(커버리지 임계치, 정적 분석 통과, 테스트 통과율)이에요. 테스트 실패가 빌드를 막지 못하면 그 자동화는 장식일 뿐이에요.

Jenkins

Jenkins는 빌드, 테스트, 배포를 자동 실행하는 오픈소스 자동화 서버예요. 파이프라인을 코드로 적은 Jenkinsfile을 저장소에 함께 두고 버전 관리해요(Pipeline as Code).

pipeline {
    agent any
    stages {
        stage('Build') {
            steps { sh 'make build' }
        }
        stage('Test') {
            steps { sh 'pytest' }
        }
        stage('Deploy') {
            steps { echo 'Deploying...' }
        }
    }
}

전체를 pipeline으로 감싸고, agent로 실행 환경을 정하고, stages 안에 stage를, 각 stage 안에 steps를 둬요. 코드 푸시나 일정으로 자동 실행하고, 실패 시 빌드를 멈추고 알림을 보내요.

Jenkins vs GitHub Actions

구분특징
Jenkins자체 서버를 설치/운영, 플러그인이 매우 풍부. 커스터마이즈에 강함. 설정은 Jenkinsfile(Groovy).
GitHub ActionsGitHub에 내장된 클라우드형, 서버 관리 불필요. 저장소와 긴밀. 설정은 YAML 파일.

한눈에 정리

  • CI/CD : 자주 합쳐 자동 빌드/테스트(CI), 통과하면 자동 배포(CD).
  • 테스트 배치 : 커밋엔 빠른 단위, 릴리스엔 전체 회귀/E2E.
  • 품질 게이트 : 실패가 빌드를 막아야 의미가 있어요.
  • Jenkins : Jenkinsfile로 pipeline/agent/stages/steps 구성.

채용 공고로 보는 요구 역량

공고의 자격요건과 우대사항을 읽으면 무엇을 먼저 준비할지 보여요. QA 채용 공고에서 실제로 자주 요구하는 것들을 정리했어요. (회사와 연차에 따라 편차가 커요.)

ISTQB 자격증, 솔직한 가치

있으면 가산, 없다고 탈락은 아니에요. 한국 공고엔 ISTQB 언급이 흔하지만 대개 우대사항이에요. 신입에겐 필수가 아니고, 이론 학습 가이드와 변별력 정도의 가치예요. 자격증보다 직접 만든 테스트 산출물(포트폴리오)이 우선이에요.

주니어가 먼저 챙길 핵심 5가지

  • 1. 테스트 케이스/시나리오 작성 (모든 공고의 1순위).
  • 2. 버그 리포트 작성(재현 절차, 기대/실제, 로그) + 꼼꼼함.
  • 3. 테스트 기본 이론(레벨/유형, SDLC, 7원칙).
  • 4. Jira 등 협업툴 + 애자일/스크럼 흐름.
  • 5. 포트폴리오(직접 만든 TC/버그리포트 공개) + 기초 기술(SQL, API). 자동화 지망이면 언어 1개와 Selenium/Playwright 추가.
현실 한 줄. 신입 QA 채용 자체가 드문 편이에요. 자격증보다 "직접 만든 테스트 산출물"이 차별화 요인이고, 자동화/SDET를 노린다면 코딩(언어 1개와 프레임워크)이 사실상 진입 요건이에요.
