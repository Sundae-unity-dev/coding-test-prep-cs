Coding Test Prep Guide

엘리스 트랙에서 익힌 언어로 코딩 테스트를 준비하는 분들을 위해 응시 전략부터 체크리스트까지 한 파일에 담았어요. 필요한 부분만 골라 읽어도 괜찮아요.

📚 필수 개념 정리로 이동 → ✍️ 예시 문제 풀어보기

1코딩 테스트, 무엇을 평가하나요?

코딩 테스트는 "어려운 알고리즘을 아는가"보다 주어진 문제를 제한 시간 안에 정확하게 구현할 수 있는가를 봐요. 그래서 화려한 풀이보다 다음 세 가지가 훨씬 중요해요.

  • 정확성 : 모든 테스트 케이스를 통과하는 코드인지. 엣지 케이스(입력의 최소값, 최대값, 빈 입력)까지 챙겼는지가 당락을 가르는 경우가 많아요.
  • 효율성 : 입력 크기가 클 때 시간 제한 안에 도는지. 정답 로직이어도 시간 초과면 0점이에요.
  • 구현 속도 : 아는 것을 빠르게 코드로 옮기는 손. 이건 순수하게 연습량에 비례해요.
핵심 한 줄 : 코딩 테스트 합격선은 보통 "전체 문제의 60~70% 정확히 풀기"예요. 어려운 한 문제에 매달리기보다 풀 수 있는 문제를 확실히 다 맞히는 전략이 유리해요.

C#으로 응시해도 괜찮을까요?

네, 괜찮아요. 프로그래머스를 비롯한 주요 플랫폼이 C#을 지원하고, 여러분이 엘리스 트랙에서 이미 익힌 문법을 그대로 쓸 수 있다는 게 큰 장점이에요. 다만 두 가지는 미리 확인해 두세요.

  • 지원 언어 목록에 C#이 있는지 응시 전에 반드시 확인하세요. 회사 자체 시험 중에는 C#을 빼는 곳도 있어요.
  • 플랫폼마다 .NET 버전이 달라요. PriorityQueue 같은 최신 기능은 시험 환경에서 한 번 컴파일해 보고 쓰는 게 안전해요.

2단계별 학습 로드맵 (16주 빌드업)

코딩 테스트 대비는 한 번에 끝나지 않아요. 입문 → 기초 → 중급 → 심화 네 단계로, 각 4주씩 약 16주에 걸쳐 차곡차곡 쌓는 걸 추천해요. 시간이 부족하면 각 단계를 비율대로 줄이되 순서는 지키세요. 아래 진단으로 내 시작 지점부터 정해 보세요.

내 시작 단계 진단

아래 4개에 예/아니오로 답하면 어디부터 시작하면 좋을지 알려드려요.

1. C# 입출력, 문자열, List와 Dictionary를 막힘없이 다룰 수 있어요
2. 정렬, 이분 탐색, DFS와 BFS를 직접 구현할 수 있어요
3. 그리디와 DP로 점화식을 세워 풀 수 있어요
4. 다익스트라, Union-Find 같은 그래프와 심화 유형도 익숙해요
입문
1~4주, 언어와 기초 자료구조언어 문법과 기본 자료구조를 손에 붙여 "문제를 코드로 옮기기" 자체에 익숙해지는 단계예요.
개념 0/8, 문제 0/8
개념 시간 복잡도 배열과 문자열 스택과 큐 해시
문제 몬스터 사냥 턴 수 전리품 골드 합산 최고 점수 차이 파티 평균 레벨 주문서의 룬 세기 거꾸로 외우는 주문 보물 지도 칸 세기 짝수 마나석 합
졸업 조건 입출력과 문자열 처리가 손에 익고, 위 입문 문제들을 풀이 없이 스스로 풀 수 있으면 다음 단계로.
주차별 계획 보기
  • 1주 입출력 빠른 템플릿, 시간 복잡도 감각, 배열과 문자열(StringBuilder)
  • 2주 스택과 큐, 괄호/대기열 패턴
  • 3주 해시(Dictionary로 세기, HashSet로 존재 확인)
  • 4주 쉬운 문제 양치기(하루 3~5문제), 1~3주 복습
기초
5~8주, 핵심 알고리즘코딩 테스트의 절반을 차지하는 정렬, 탐색, 그래프 순회를 유형별로 익혀요.
개념 0/8, 문제 0/8
개념 정렬 이분 탐색 완전 탐색 DFS와 BFS
문제 스킬 콤보 판정 연속 버프 정리 돌아오지 못한 모험가 재료 조합 도감 NPC 퀴즈 찍기 구간 랭킹 보드 최강 전투력 여분 무기 빌려주기
졸업 조건 커스텀 정렬 기준을 세우고, BFS로 최단 거리를, 재귀로 완전 탐색을 막힘없이 짤 수 있으면 통과.
주차별 계획 보기
  • 5주 정렬(커스텀 비교), 이분 탐색(경계 잡기)
  • 6주 완전 탐색과 백트래킹(순열/조합)
  • 7주 DFS와 BFS, 격자/그래프 순회
  • 8주 약한 유형 보강, 틀린 문제 다시 풀기
중급
9~12주, 자주 나오는 패턴점화식과 그리디 증명, 자료구조를 활용한 최적화로 한 단계 어려운 문제에 도전해요.
개념 0/10, 문제 0/8
개념 그리디 동적 계획법 우선순위 큐 투 포인터와 누적 합 구현과 시뮬레이션
문제 강화 등급 집계 골드 피라미드 구간 보상 합 연속 사냥 구간 포션 연성 던전 최단 탈출 마나 부호 조합 패치 배포 대기열
졸업 조건 "dp[i]가 무엇인지" 문장으로 정의하고, 그리디가 통하는 이유를 한 문장으로 설명할 수 있으면 통과.
주차별 계획 보기
  • 9주 그리디(정렬 후 선택, 통하는 이유 설명)
  • 10주 DP(상태 정의 → 점화식 → 초기값)
  • 11주 우선순위 큐, 투 포인터와 누적 합
  • 12주 구현/시뮬레이션, 중급 종합 모의 1회
심화
13~16주, 그래프 심화와 수학, 실전가중치 그래프, 분리 집합, 비트마스킹, 수학까지 채우고 실전 감각으로 마무리해요.
개념 0/14, 문제 0/8
개념 다익스트라 Union-Find 비트마스킹 위상 정렬 길찾기(A*) 소수와 체 유클리드 호제법
문제 포탈 검문소 성채 최단 보급로 미로 최소 마나 탈출 길드 동맹 세기 연구 해금 순서 원정대 보급 배낭 스킬 조합 최대 효율 톱니바퀴 동기화
졸업 조건 문제 유형을 보고 알고리즘을 30초 안에 떠올리고, 시간 제한 안에 구현을 마칠 수 있으면 실전 준비 완료.
주차별 계획 보기
  • 13주 다익스트라, 위상 정렬(가중치/의존 그래프)
  • 14주 Union-Find, 비트마스킹
  • 15주 수학(소수와 체, 유클리드 호제법), 구현 심화
  • 16주 실전 모의고사 2~3회, 약점 마무리

3문제를 읽는 법 : 제약 조건이 답을 알려줘요

문제를 받으면 본문보다 먼저 입력 크기 N의 범위와 시간 제한을 보세요. 대부분의 채점 환경은 1초에 약 1억 번(108) 정도의 단순 연산을 처리한다고 보면 돼요. 여기서 어떤 시간 복잡도까지 허용되는지가 바로 계산되고, 그게 곧 쓸 수 있는 알고리즘의 후보가 돼요.

N의 최대 크기허용되는 복잡도후보 알고리즘
N ≤ 10O(N!)순열 완전 탐색 (10! ≈ 360만)
N ≤ 20O(2N)부분집합 탐색, 비트마스킹
N ≤ 500O(N3)플로이드 워셜, 3중 반복문
N ≤ 2,000O(N2)2중 반복문, 기초 DP
N ≤ 100,000O(N log N)정렬, 이분 탐색, 우선순위 큐
N ≤ 10,000,000O(N)투 포인터, 누적 합, 해시
그보다 크면O(log N) 또는 O(1)수학적 규칙 찾기, 이분 탐색
예시 : N이 최대 100,000인데 2중 반복문을 쓰면 100,000 x 100,000 = 1010번 연산이라 무조건 시간 초과예요. 이 경우 "정렬해서 이분 탐색" 또는 "해시로 조회"처럼 O(N log N) 이하의 방법을 찾아야 해요.

풀이 순서 5단계

  • 1단계 : 제약 조건 확인 → 허용 복잡도 계산 (위 표)
  • 2단계 : 입출력 예시를 손으로 따라가며 문제를 정확히 이해 (예시가 왜 그 답이 되는지 설명할 수 있어야 해요)
  • 3단계 : 풀이를 한국어로 먼저 적기 ("정렬한 다음, 양 끝에서 포인터를 좁혀가며...")
  • 4단계 : 코드로 옮기기. 막히면 3단계 문장으로 돌아가기
  • 5단계 : 예시 입력 + 직접 만든 엣지 케이스(최소 입력, 최대 입력, 같은 값 반복)로 검증

4시간 배분 전략

예를 들어 2시간에 4문제가 나오는 시험이라면, 문제당 30분이 아니라 아래처럼 움직이는 게 좋아요.

  • 처음 10분 : 4문제를 전부 훑고 난이도 순서를 정해요. 쉬운 문제부터 푸는 게 원칙이에요.
  • 쉬운 문제(1~2번) : 각 20분 안에 끝내고 제출까지 완료해요. 여기서 시간을 벌어야 해요.
  • 중간 문제 : 30~40분. 막히면 일단 부분 점수라도 받을 수 있는 코드를 제출해 두세요.
  • 어려운 문제 : 남은 시간 전부. 단, 종료 15분 전에는 손을 떼고 푼 문제들을 재점검해요.
30분 규칙 : 한 문제에서 30분 동안 진전이 없으면 무조건 다음 문제로 넘어가세요. "조금만 더 하면 풀릴 것 같은데"가 가장 위험한 함정이에요. 다른 문제를 풀다 보면 머리가 환기되어 돌아왔을 때 풀리는 경우가 많아요.
제출 타이밍 : 다 풀고 한 번에 제출하지 말고, 한 문제가 끝날 때마다 바로 제출해서 채점 결과를 확인하세요. 틀렸다면 일찍 알수록 고칠 시간이 생겨요.

5C# 코딩 테스트 기본 세팅

시험장에서 바로 타이핑할 수 있도록 기본 틀을 외워두면 좋아요. 특히 입출력이 많은 문제에서는 Console.ReadLine과 Console.WriteLine을 그대로 쓰면 그것만으로 시간 초과가 날 수 있어서, 아래 빠른 입출력 틀을 기본값으로 쓰는 걸 권해요.

빠른 입출력 템플릿

using System;
using System.IO;
using System.Text;
using System.Collections.Generic;
using System.Linq;

class Program
{
    static void Main()
    {
        // 버퍼를 사용하는 빠른 입력기예요
        var input = new StreamReader(Console.OpenStandardInput());
        // 출력은 StringBuilder에 모았다가 마지막에 한 번만 내보내요
        var sb = new StringBuilder();

        int n = int.Parse(input.ReadLine());

        for (int i = 0; i < n; i++)
        {
            // 공백으로 구분된 한 줄을 숫자 배열로 받는 패턴이에요
            // 공백이 여러 개여도 안전하도록 빈 항목을 제거해요
            int[] arr = input.ReadLine().Split(new[] { ' ' }, StringSplitOptions.RemoveEmptyEntries).Select(int.Parse).ToArray();
            sb.AppendLine((arr[0] + arr[1]).ToString());
        }

        Console.Write(sb);
    }
}

실제 시험에서 점수를 깎아 먹는 건 어려운 알고리즘이 아니라 이런 실수들이에요. 하나씩 열어서 확인해 보세요.

1. int 범위 초과 (오버플로)

int의 최대값은 약 21억이에요. "N이 최대 100,000이고 각 값이 최대 100,000"이면 합이나 곱이 21억을 쉽게 넘어요. 합계, 곱셈, 누적값을 다룰 때는 long을 기본으로 쓰세요.

long sum = 0;
foreach (var v in arr) sum += v;  // int sum이면 조용히 틀려요
2. 반복문 안에서 문자열 + 연결

C#의 string은 불변이라 s += "..."를 반복하면 매번 새 문자열을 만들어요. 반복 횟수가 많으면 시간 초과의 주범이에요. 반복해서 이어 붙일 때는 반드시 StringBuilder를 쓰세요.

3. Console.WriteLine을 수만 번 호출

출력이 많은 문제에서 Console.WriteLine을 반복 호출하면 출력만으로 시간 초과가 날 수 있어요. StringBuilder에 모아서 마지막에 Console.Write(sb) 한 번으로 내보내세요.

4. List에서 반복 중 요소 삭제

foreach 중에 컬렉션을 수정하면 예외가 나고, for문으로 앞에서부터 지우면 인덱스가 밀려요. 지울 때는 뒤에서 앞으로 돌거나, 조건에 맞는 것만 남긴 새 리스트를 만드는 게 안전해요.

// 안전한 방법 1 : 뒤에서부터
for (int i = list.Count - 1; i >= 0; i--)
    if (list[i] % 2 == 0) list.RemoveAt(i);

// 안전한 방법 2 : 새로 거르기
list = list.Where(v => v % 2 != 0).ToList();
5. 정수 나눗셈 함정

int / int는 결과도 int라서 소수점이 버려져요. 5 / 2는 2.5가 아니라 2예요. 실수 결과가 필요하면 한쪽을 (double)로 캐스팅하세요. 반대로 "내림이 필요한" 문제에서는 정수 나눗셈이 오히려 정답 도구예요.

6. 엣지 케이스 누락

제출 전에 항상 이 네 가지를 자문하세요.

  • N = 1 또는 빈 입력일 때도 동작하나요?
  • 모든 값이 같을 때는요?

  • 음수나 0이 입력에 포함될 수 있나요?
7. 재귀 깊이 초과 (스택 오버플로)

DFS를 재귀로 짰는데 정점이 수십만 개면 스택이 터질 수 있어요. C#은 기본 스택 크기가 작은 편이라, 깊은 재귀가 예상되면 Stack<T>를 쓴 반복문 DFS로 바꾸는 게 1순위예요. 아래처럼 스택 크기를 키운 스레드에서 실행하는 방법도 있는데, 이건 Main을 직접 작성하는 환경에서만 쓸 수 있어요. 함수 형태로 제출하는 채점 환경에서는 반복문 DFS를 쓰세요.

// 스택 크기를 256MB로 키운 스레드에서 실행하는 패턴이에요
var thread = new Thread(Solve, 256 * 1024 * 1024);
thread.Start();
thread.Join();
8. 문제를 끝까지 안 읽고 코딩 시작

"출력 형식" 문단에 함정이 숨어 있는 경우가 많아요. 소수점 자리수, 줄바꿈 여부, 공백 하나 차이로도 오답 처리돼요. 출력 예시와 내 출력을 글자 단위로 비교하는 습관을 들이세요.
