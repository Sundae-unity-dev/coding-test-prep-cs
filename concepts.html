기초

시간 복잡도 (빅오 표기법)

입력이 커질 때 연산 횟수가 얼마나 빨리 늘어나는지를 나타내는 표기예요. 코딩 테스트의 모든 판단이 여기서 시작돼요.

왜 알아야 하나요?

채점 서버는 1초에 약 1억 번(108) 정도의 단순 연산을 처리한다고 보면 돼요. 내가 짠 코드의 연산 횟수를 어림해서 이 한계 안에 들어오는지 미리 판단해야, 코드를 다 짜고 나서 시간 초과를 받는 일을 막을 수 있어요.

복잡도이름N = 100,000일 때 연산 수대표 예
O(1)상수1배열 인덱스 접근, Dictionary 조회
O(log N)로그약 17이분 탐색
O(N)선형100,000배열 한 바퀴 순회
O(N log N)선형 로그약 170만좋은 정렬 알고리즘
O(N²)제곱100억 (시간 초과!)2중 반복문
O(2ᴺ)지수측정 불가모든 부분집합 탐색

복잡도 읽는 법

  • 반복문 하나가 N번 돌면 O(N), 반복문 안에 반복문이 또 있으면 O(N²)이에요.
  • 가장 큰 항만 남겨요. O(N² + N)은 그냥 O(N²)이라고 써요.
  • 상수는 무시해요. O(2N)도 O(N)이에요.
  • "반씩 줄여가며 찾는" 구조가 보이면 O(log N)이에요.

숨어 있는 반복문을 조심하세요

C#의 편리한 메서드 중 상당수는 내부적으로 전체를 순회해요. 반복문 안에서 무심코 쓰면 복잡도가 한 단계 올라가요.

// list.Contains는 내부에서 N번 비교해요 (O(N))
// 이걸 반복문 안에서 쓰면 전체가 O(N * N)이 돼요
foreach (var v in queries)
{
    if (list.Contains(v)) count++;   // 위험!
}

// HashSet.Contains는 평균 O(1)이라 전체가 O(N)으로 끝나요
var set = new HashSet<int>(list);
foreach (var v in queries)
{
    if (set.Contains(v)) count++;    // 안전!
}
자주 하는 실수 : ① 문제의 N 최대값을 안 보고 코딩을 시작해 다 짠 뒤 시간 초과를 만나요. 항상 제약부터 읽고 허용 복잡도를 역산하세요. ② 반복문 안의 list.Contains, list.IndexOf, 문자열 += 누적 같은 "숨은 O(N)"을 못 보고 넘어가요. ③ 시간뿐 아니라 재귀 깊이와 메모리도 복잡도의 일부예요.

한눈에 정리

  • 언제 : 코드를 짜기 전에 제한 시간 안에 도는지 가늠할 때예요.
  • 복잡도 : 입력 N과 제한 시간으로 허용 복잡도를 역산해요. 1초가 대략 1억 연산이에요.
  • 핵심 : 반복문 중첩 깊이와 입력 크기로 빅오를 읽어요.
  • 실수 주의 : 상수와 낮은 차수는 무시하되, N이 작으면 O(2^N)도 통과한다는 점을 기억하세요.
QUIZ 1N이 최대 100,000인 문제에서 시간 안에 통과할 가능성이 가장 높은 복잡도는 무엇일까요?
정답: O(N log N)
N = 100,000일 때 O(N log N)은 약 170만 번 연산이라 여유 있게 통과해요. O(N²)은 100억 번이라 시간 초과예요. 제약 조건에서 허용 복잡도를 역산하는 이 감각이 문제 풀이의 출발점이에요.
QUIZ 2다음 코드의 시간 복잡도는 무엇일까요?
for (int i = 0; i < n; i++)
    for (int j = i + 1; j < n; j++)
        Check(i, j);
정답: O(N²)
안쪽 반복이 i에 따라 줄어들지만 총 횟수는 N(N-1)/2, 즉 약 N²/2예요. 상수(1/2)는 무시하므로 O(N²)이에요. "절반만 도니까 O(N log N)"이라고 착각하기 쉬운 패턴이에요.

가장 많이 출제되는 영역이에요. 화려한 알고리즘 없이 배열과 문자열을 정확히 다루는 것만으로 풀리는 문제가 전체의 30% 이상이에요.

문자열은 불변이에요

C#의 string은 한 번 만들면 내용을 바꿀 수 없어요. s += "a"는 기존 문자열을 수정하는 게 아니라 매번 새 문자열을 통째로 복사해요. 반복문에서 이어 붙이면 O(N²)이 되니, 반복 연결에는 반드시 StringBuilder를 쓰세요.

// 느려요 : 반복마다 전체 복사가 일어나요
string s = "";
for (int i = 0; i < n; i++) s += i;

// 빨라요 : 내부 버퍼에 덧붙이기만 해요
var sb = new StringBuilder();
for (int i = 0; i < n; i++) sb.Append(i);
string result = sb.ToString();

자주 쓰는 도구 모음

string s = "hello world";

s.Split(' ')                 // 공백 기준으로 쪼개기 -> ["hello", "world"]
s.Substring(0, 5)            // 0번째부터 5글자 -> "hello"
s.IndexOf("world")           // 위치 찾기 -> 6 (없으면 -1)
s.Replace("l", "L")          // 치환 -> "heLLo worLd"
s.ToUpper()                  // 대문자로
s.Trim()                     // 양쪽 공백 제거
char.IsDigit(s[0])           // 숫자인지 판별
new string(s.Reverse().ToArray())  // 뒤집기

직접 풀어보기

문제 : 문자열 두 개가 애너그램(서로 글자를 재배열한 관계, 예: "listen"과 "silent")인지 판별하는 함수를 작성해 보세요.

static bool IsAnagram(string a, string b)
{
    // 길이가 다르면 바로 탈락이에요
    if (a.Length != b.Length) return false;

    // 두 문자열을 글자 단위로 정렬해서 같은지 비교해요
    var sa = a.OrderBy(c => c).ToArray();
    var sb = b.OrderBy(c => c).ToArray();
    return sa.SequenceEqual(sb);
}

정렬 방식은 O(N log N)이에요. 글자별 개수를 배열로 세면 O(N)으로도 풀 수 있어요. 둘 다 면접에서 설명할 수 있으면 좋아요.

한눈에 정리

  • 언제 : 순서가 있는 데이터를 다루는 거의 모든 문제의 기본 도구예요.
  • 복잡도 : 인덱스 접근 O(1), 끝에 추가 O(1) 분할상환, 중간 삽입과 삭제는 O(N).
  • 핵심 : 문자열은 불변이라 자주 이어 붙일 때는 StringBuilder를 써요.
  • 실수 주의 : 반복문 안에서 문자열을 + 로 계속 이으면 전체 O(N^2)이 되어 느려져요.

정답: StringBuilder
string은 불변이라 += 마다 전체 복사가 일어나 O(N²)이 돼요. StringBuilder는 내부 버퍼에 덧붙이는 구조라 O(N)이에요. 출력이 많은 문제에서는 필수예요.

"넣은 순서"를 다루는 두 가지 기본 자료구조예요. 괄호 검사, 작업 대기열, BFS 등 활용처가 정말 많아요.

대표 문제 : 괄호 짝 검사

여는 괄호를 만나면 스택에 넣고, 닫는 괄호를 만나면 스택 맨 위와 짝이 맞는지 확인하는 패턴이에요. 스택 문제의 절반은 이 변형이에요.

static bool IsValid(string s)
{
    var stack = new Stack<char>();
    foreach (char c in s)
    {
        if (c == '(')
        {
            stack.Push(c);            // 여는 괄호는 일단 쌓아둬요
        }
        else // 닫는 괄호일 때
        {
            // 짝이 될 여는 괄호가 없으면 잘못된 문자열이에요
            if (stack.Count == 0) return false;
            stack.Pop();              // 짝을 맞췄으니 하나 제거해요
        }
    }
    // 다 돌고 나서 스택이 비어 있어야 모든 짝이 맞은 거예요
    return stack.Count == 0;
}

  • 큐 : "들어온 순서대로 처리", "대기열", "한 칸씩 퍼져 나가기"(BFS) 같은 표현이 보일 때

한눈에 정리

  • 언제 : 짝 맞추기와 되돌리기는 스택, 순서대로 처리와 BFS는 큐예요.
  • 복잡도 : push, pop, enqueue, dequeue 모두 O(1).
  • 핵심 : 괄호와 수식, DFS는 스택, 레벨 순회와 최단 거리는 큐로 풀어요.
  • 실수 주의 : 비어 있을 때 Peek 또는 Pop 은 예외가 나니 Count 를 먼저 확인하세요.

해시 (Dictionary와 HashSet)

"찾기"를 평균 O(1)로 만들어주는 자료구조예요. 시간 초과가 나는 풀이를 통과하는 풀이로 바꾸는 가장 흔한 열쇠예요.

두 가지 도구

Dictionary<K, V>HashSet<T>
저장하는 것키와 값의 쌍값만 (중복 자동 제거)
대표 용도이름별 점수, 단어별 등장 횟수"이미 본 값인가?" 판별, 중복 제거
조회 속도평균 O(1)평균 O(1)

패턴 1 : 등장 횟수 세기

var count = new Dictionary<string, int>();
foreach (var word in words)
{
    // TryGetValue는 키가 없을 때 예외 대신 false를 줘요
    count[word] = count.TryGetValue(word, out int c) ? c + 1 : 1;
}

// 가장 많이 나온 단어 찾기
string best = count.OrderByDescending(kv => kv.Value).First().Key;
주의 : 존재하지 않는 키를 count["없는키"]로 읽으면 KeyNotFoundException이 나요. 읽기 전에 ContainsKey로 확인하거나 TryGetValue를 쓰세요.

패턴 2 : 중복 제거와 존재 확인

// 중복 제거 : 배열을 넣기만 하면 끝나요
var unique = new HashSet<int>(arr);

// "두 수의 합이 target이 되는 쌍이 있는가?" - 해시의 대표 활용이에요
static bool HasPairSum(int[] arr, int target)
{
    var seen = new HashSet<int>();
    foreach (var v in arr)
    {
        // 지금 값의 짝(target - v)을 이전에 본 적 있는지 O(1)로 확인해요
        if (seen.Contains(target - v)) return true;
        seen.Add(v);
    }
    return false;
}
이 발상이 핵심이에요 : 2중 반복문 O(N²)으로 모든 쌍을 검사하는 대신, "지금까지 본 것"을 해시에 기록하며 한 번만 순회하면 O(N)이 돼요. 시간 초과가 나면 가장 먼저 "해시로 조회를 O(1)로 만들 수 없나?"를 자문하세요.

HasPairSum 따라가기 : arr = [3, 5, 2, 8], target = 10

82있음 → true(2 + 8 = 10)

각 값에서 "내 짝을 전에 봤나?"만 O(1)로 묻고 지나가요. 전체를 딱 한 번만 훑어 O(N)에 끝나죠.

  • 이럴 때 써요
  • "~가 있는지/몇 번 나오는지/중복인지" 조회가 반복될 때 (O(1) 조회)
  • 2중 반복문 O(N²)이 시간 초과날 때, 한쪽을 해시로 바꿔 O(N)으로
  • "이름별, 값별로 묶어서 세기"가 필요할 때 (Dictionary)

한눈에 정리

  • 언제 : 빠른 존재 확인, 개수 세기, 중복 제거가 필요할 때예요.
  • 복잡도 : 평균 O(1) 로 넣고 찾아요.
  • 핵심 : 개수가 필요하면 Dictionary, 존재 여부만이면 HashSet 을 써요.
  • 실수 주의 : 없는 키를 바로 접근하면 예외가 나요. TryGetValue 나 ContainsKey 로 막으세요.
QUIZ 1참가자 명단과 완주자 명단이 주어지고 "완주하지 못한 한 사람"을 찾아야 해요 (동명이인 존재 가능). 알맞은 도구는 무엇일까요?
정답: Dictionary
동명이인이 있으면 HashSet은 같은 이름을 하나로 합쳐버려서 못 풀어요. 이름을 키, 인원수를 값으로 세고 완주자에서 하나씩 빼면 1이 남는 이름이 답이에요. 실제 기출(프로그래머스 "완주하지 못한 선수")의 핵심 함정이에요.
QUIZ 2Dictionary에서 존재하지 않는 키를 대괄호로 읽으면(dict["x"]) 어떻게 될까요?
정답: 예외가 발생한다
C#의 Dictionary는 없는 키를 읽으면 즉시 예외를 던져요. 런타임 에러로 0점이 되는 흔한 원인이라, ContainsKey 확인이나 TryGetValue 사용을 습관으로 만들어야 해요. 참고로 쓰기(dict["x"] = 1)는 키가 없어도 새로 만들어줘서 괜찮아요.
핵심 알고리즘

정렬

코딩 테스트에서는 정렬 알고리즘을 직접 구현하기보다 내장 정렬을 자유자재로 쓰는 능력이 중요해요. 특히 커스텀 기준 정렬이 단골이에요.

기본 사용법

int[] arr = { 5, 2, 8, 1 };

Array.Sort(arr);                            // 오름차순 : 1 2 5 8
Array.Sort(arr, (a, b) => b.CompareTo(a));  // 내림차순 : 8 5 2 1

var list = new List<int> { 5, 2, 8 };
list.Sort();                                // List도 같은 방식이에요

내장 정렬의 복잡도는 O(N log N)이에요. 비교 함수에서 a.CompareTo(b)는 a가 작으면 음수를 반환하고, 음수면 a가 앞으로 와요. "음수 = 그대로, 양수 = 자리 바꿈"으로 기억하세요.

커스텀 기준 정렬 (출제 빈도 최상)

"점수가 높은 순으로, 점수가 같으면 이름 사전순으로" 같은 다중 기준이 단골이에요.

var students = new List<(string name, int score)>
{
    ("kim", 90), ("lee", 85), ("park", 90)
};

students.Sort((a, b) =>
{
    // 1순위 : 점수 내림차순 (b가 먼저 오도록 b.CompareTo(a))
    if (a.score != b.score) return b.score.CompareTo(a.score);
    // 2순위 : 이름 오름차순
    return string.Compare(a.name, b.name);
});
// 결과 : kim(90), park(90), lee(85)

LINQ 방식도 알아두세요

// OrderBy(오름차순) + ThenBy(2차 기준)를 사슬처럼 이어요
var sorted = students
    .OrderByDescending(s => s.score)
    .ThenBy(s => s.name)
    .ToList();
정렬 안정성 : 내장 Array.Sort와 List.Sort는 같은 값의 원래 순서를 보장하지 않는 불안정 정렬이에요. 반대로 LINQ OrderBy/ThenBy는 안정 정렬이라 동점이면 입력 순서를 지켜요. 동점의 순서가 중요한 문제라면 OrderBy를 쓰거나, 비교 기준을 동점까지 끝까지 명시하세요.
숫자를 문자열처럼 정렬하는 함정 : 문자열 "10"은 "9"보다 사전순으로 앞이에요 ("1" < "9"). 숫자 크기로 정렬하려면 int.Parse로 변환한 뒤 비교해야 해요. "가장 큰 수 만들기" 유형에서 자주 등장하는 함정이에요.

  • "가장 큰/작은", "최대 몇 개", "겹치지 않게" 같은 표현 → 정렬해 두면 그리디나 투 포인터로 이어져요
  • "무엇을 기준으로 줄 세우면 답이 단순해질까?"가 보이는 문제 (정렬은 거의 모든 유형의 전처리)
  • 정렬 후 이분 탐색을 쓰려고 할 때 (먼저 정렬해야 탐색이 성립)

한눈에 정리

  • 언제 : 비교, 그리디, 이분 탐색의 전처리로 거의 항상 쓰여요.
  • 복잡도 : O(N log N).
  • 핵심 : 기준이 여러 개면 커스텀 비교(첫 기준이 같을 때 둘째 기준)를 만들어요.
  • 실수 주의 : 합 비교에서 int 오버플로, 문자열 숫자의 사전순 함정을 조심하세요.

이분 탐색

정렬된 범위에서 "절반씩 잘라가며" 찾는 기법이에요. 10억 개 중에서도 30번 만에 찾아요 (O(log N)).

동작 원리

업다운 게임을 떠올리면 돼요. 1부터 100 사이의 숫자를 맞힐 때 50부터 부르고, "업"이면 75, "다운"이면 25를 부르죠. 매번 후보가 절반으로 줄어드는 이 구조가 이분 탐색이에요. 전제 조건은 데이터가 정렬되어 있어야 한다는 것이에요.

// 정렬된 배열에서 target의 위치를 찾아요. 없으면 -1을 반환해요
static int BinarySearch(int[] arr, int target)
{
    int lo = 0, hi = arr.Length - 1;
    while (lo <= hi)
    {
        // (lo + hi) / 2는 두 값이 크면 오버플로될 수 있어 이렇게 써요
        int mid = lo + (hi - lo) / 2;

        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) lo = mid + 1;  // 오른쪽 절반으로
        else hi = mid - 1;                          // 왼쪽 절반으로
    }
    return -1;
}

C#에는 Array.BinarySearch(arr, target)이 내장되어 있어요. 찾으면 인덱스를, 없으면 "들어갈 위치"를 비트 반전한 음수를 반환해요. 그래서 int pos = ~result; 로 삽입 위치를 구할 수 있어요. 정렬이 안 된 배열에선 결과가 보장되지 않으니 주의하세요. 다만 직접 구현도 손에 익혀두세요. 변형 문제에서는 내장 함수로 부족한 경우가 많아요.

한 단계 위 : 파라메트릭 서치

"최소 시간", "최대 길이"처럼 답 자체를 이분 탐색하는 변형이에요. "답이 X라면 조건을 만족하는가?"라는 예/아니오 판정 함수를 만들고, X를 이분 탐색으로 좁혀가요.

// 예 : 장비 N개로 M명을 심사하는 데 필요한 최소 시간
// "시간 t 안에 M명 이상 심사 가능한가?"를 판정하며 t를 좁혀요
long lo = 1, hi = 1_000_000_000_000;
while (lo < hi)
{
    long mid = lo + (hi - lo) / 2;
    if (CanFinish(mid)) hi = mid;      // 가능하면 더 짧은 시간 도전
    else lo = mid + 1;                 // 불가능하면 시간을 늘려요
}
// 종료 시 lo가 조건을 만족하는 최소값이에요
이럴 때 써요 : "최소값의 최대화", "최대값의 최소화", 답의 범위가 10억을 넘는 문제는 파라메트릭 서치를 의심하세요.

한눈에 정리

  • 언제 : 정렬된 데이터에서 위치를 찾거나, 답을 정해두고 가능 여부를 판정하는 파라메트릭 서치에 써요.
  • 복잡도 : O(log N).
  • 핵심 : lo 와 hi 경계, 그리고 종료 조건을 명확히 정해요.
  • 실수 주의 : mid 계산과 경계 갱신을 잘못하면 무한 루프나 한 칸 차이(off-by-one)가 생겨요.

완전 탐색 (브루트 포스)

가능한 모든 경우를 빠짐없이 검사하는 방법이에요. 입력이 작은 문제에서는 가장 확실한 정공법이에요.

이럴 때 써요

시간 복잡도 표를 다시 떠올려 보세요. 경우의 수가 약 1억 개 이하면 완전 탐색이 통과해요. N이 20 이하(부분집합 2²⁰ ≈ 백만), 10 이하(순열 10! ≈ 360만)면 거의 확실히 완전 탐색 문제예요.

재귀로 순열 만들기

"N개 중 몇 개를 골라 줄 세우는" 문제의 뼈대예요. 이 틀을 외워두면 변형이 자유로워요.

static int[] arr = { 1, 2, 3 };
static bool[] used = new bool[3];
static List<int> current = new();

static void Permute()
{
    // 다 골랐으면 하나의 순열이 완성된 거예요
    if (current.Count == arr.Length)
    {
        Console.WriteLine(string.Join(" ", current));
        return;
    }
    for (int i = 0; i < arr.Length; i++)
    {
        if (used[i]) continue;   // 이미 쓴 숫자는 건너뛰어요

        used[i] = true;          // 1. 선택하고
        current.Add(arr[i]);
        Permute();               // 2. 다음 자리로 들어갔다가
        current.RemoveAt(current.Count - 1);
        used[i] = false;         // 3. 돌아와서 선택을 되돌려요 (백트래킹)
    }
}
백트래킹이란? : 위 코드의 "선택 → 재귀 → 되돌리기" 구조예요. 여기에 "이 길은 더 가봐야 답이 안 된다"는 조건(가지치기)을 추가해서 일찍 돌아 나오면, 완전 탐색보다 훨씬 빨라져요. N-Queen 문제가 대표적이에요.

조합은 시작 인덱스로

순서가 상관없는 "조합"은 used 배열 대신 시작 인덱스를 넘겨서 자기보다 뒤의 것만 고르게 하면 중복 없이 만들어져요.

static void Combine(int start, int pick)
{
    if (pick == 0) { Process(current); return; }
    for (int i = start; i < arr.Length; i++)
    {
        current.Add(arr[i]);
        Combine(i + 1, pick - 1);   // i보다 뒤에서만 고르게 해요
        current.RemoveAt(current.Count - 1);
    }
}

DFS : 영역 개수 세기

static int[,] map;          // 1 = 땅, 0 = 바다
static bool[,] visited;
static int n, m;
static int[] dx = { -1, 1, 0, 0 };
static int[] dy = { 0, 0, -1, 1 };

static void Dfs(int x, int y)
{
    visited[x, y] = true;
    for (int d = 0; d < 4; d++)
    {
        int nx = x + dx[d], ny = y + dy[d];
        // 격자 밖, 바다, 방문한 곳은 건너뛰어요
        if (nx < 0 || nx >= n || ny < 0 || ny >= m) continue;
        if (map[nx, ny] == 0 || visited[nx, ny]) continue;
        Dfs(nx, ny);
    }
}

// 모든 칸을 돌며 새 땅을 만날 때마다 섬이 하나 늘어요
int islands = 0;
for (int i = 0; i < n; i++)
    for (int j = 0; j < m; j++)
        if (map[i, j] == 1 && !visited[i, j])
        {
            islands++;
            Dfs(i, j);   // 이 섬 전체를 방문 처리해요

BFS : 최단 거리

static int Bfs(int sx, int sy)
{
    var queue = new Queue<(int x, int y)>();
    var dist = new int[n, m];        // 0이면 미방문으로 써요

    queue.Enqueue((sx, sy));
    dist[sx, sy] = 1;                // 시작 칸 포함 거리 1

    while (queue.Count > 0)
    {
        var (x, y) = queue.Dequeue();
        for (int d = 0; d < 4; d++)
        {
            int nx = x + dx[d], ny = y + dy[d];
            if (nx < 0 || nx >= n || ny < 0 || ny >= m) continue;
            if (map[nx, ny] == 0 || dist[nx, ny] != 0) continue;

            dist[nx, ny] = dist[x, y] + 1;   // 한 칸 멀어진 거리예요
            queue.Enqueue((nx, ny));
        }
    }
    return dist[n - 1, m - 1] == 0 ? -1 : dist[n - 1, m - 1];   // 도착 못 하면 -1 (게임 맵 최단거리 유형)
}
가장 흔한 버그 : 방문 표시를 "큐에서 꺼낼 때" 하면 같은 칸이 큐에 여러 번 들어가 시간 초과나 메모리 초과가 나요. 반드시 큐에 넣을 때 방문(거리) 표시를 하세요.
복잡도 : DFS도 BFS도 정점 V개와 간선 E개를 한 번씩만 보므로 O(V + E)예요. 격자 문제라면 칸 수가 곧 V, 상하좌우 연결이 E라서 사실상 O(칸 수)죠.
C# 함정 : 재귀 DFS는 정점이 수만~수십만 개로 깊어지면 호출 스택이 넘쳐 StackOverflowException이 날 수 있어요. 깊이가 깊을 것 같으면 재귀 대신 Stack<T>를 쓴 반복 DFS나 BFS로 바꾸세요.

한눈에 정리

  • 언제 : 그래프와 격자 탐색, 연결 요소 세기, 가중치 없는 최단 거리에 써요.
  • 복잡도 : O(V + E), 격자에서는 O(N x M).
  • 핵심 : 가중치 없는 최단 거리는 BFS, 깊이와 경로 탐색은 DFS 예요.
  • 실수 주의 : 방문 표시는 큐에 넣을 때 해야 같은 칸을 두 번 넣지 않아요.

그리디 (탐욕법)

매 순간 가장 좋아 보이는 선택을 반복하는 전략이에요. 통하는 문제에서는 가장 간단하고 빠른 풀이가 돼요.

그리디가 통하는 조건

"지금의 최선이 전체의 최선을 망치지 않는다"는 보장이 있어야 해요. 이 보장이 없는 문제에 그리디를 쓰면 그럴듯한 오답이 나와요. 그래서 그리디 문제의 진짜 실력은 "왜 이 선택이 항상 손해가 아닌지"를 한 문장으로 설명할 수 있는가예요.

  • 통하는 예 : 한국 동전(500, 100, 50, 10)으로 거스름돈 최소 개수 만들기. 큰 동전이 항상 작은 동전들의 배수라서 큰 것부터 쓰는 게 늘 이득이에요.
  • 안 통하는 예 : 동전이 500, 400, 100일 때 800원 거슬러주기. 그리디는 500 + 100 + 100 + 100 = 4개를 고르지만 정답은 400 + 400 = 2개예요. 이런 경우는 DP로 풀어야 해요.

대표 문제 : 회의실 배정

"회의 N개 중 겹치지 않게 최대 몇 개를 진행할 수 있는가?" 정답 전략은 끝나는 시간이 빠른 순으로 정렬해서 가능한 것부터 담는 거예요. 빨리 끝날수록 뒤에 올 회의를 위한 시간이 많이 남기 때문이에요.

// meetings : (시작, 끝) 목록
meetings.Sort((a, b) => a.end != b.end
    ? a.end.CompareTo(b.end)         // 1순위 : 끝나는 시간 빠른 순
    : a.start.CompareTo(b.start));   // 같으면 시작 시간 빠른 순

int count = 0, lastEnd = int.MinValue;
foreach (var m in meetings)
{
    // 직전 회의가 끝난 뒤에 시작하는 회의만 담아요
    if (m.start >= lastEnd)
    {
        count++;
        lastEnd = m.end;
    }
}
이럴 때 써요 : "최대한 많이", "최소 개수로" + 정렬하면 풀릴 것 같은 구조. 일단 "무엇을 기준으로 정렬할까?"부터 고민해 보세요. 그리디 문제의 80%는 정렬 기준 찾기예요.

회의 6개로 따라가기

회의 (시작, 끝) = (1,4), (3,5), (0,6), (5,7), (5,9), (8,9). 끝나는 시간 순으로 정렬한 뒤 앞에서부터 담아요.

회의 (시작, 끝)lastEnd담을까? (시작 >= lastEnd)
(1, 4)없음담음 → lastEnd = 4
(3, 5)43 < 4 라 건너뜀
(0, 6)40 < 4 라 건너뜀
(5, 7)4담음 → lastEnd = 7
(5, 9)75 < 7 라 건너뜀
(8, 9)7담음 → lastEnd = 9

결과는 3개예요. 끝이 빠른 회의를 먼저 담을수록 뒤에 더 많은 회의를 담을 여지가 생긴다는 게 이 그리디의 핵심이에요.

복잡도 : 정렬 O(N log N) + 한 번 훑기 O(N) = 전체 O(N log N)이에요. 그리디는 보통 정렬 비용이 전부예요.
자주 하는 실수 : ① 그리디가 성립하는지 확인하지 않고 "큰 것부터" 식으로 바로 적용해요. 배수 관계가 깨진 거스름돈처럼 일부 케이스만 틀리면 전제가 무너진 신호이니 DP를 의심하세요. ② 정렬 기준을 잘못 잡아요. 회의실 배정은 "끝나는 시간"이 기준인데 "시작 시간"이나 "회의 길이"로 정렬하면 틀려요. 무엇을 기준으로 정렬할지부터 정하세요. ③ 동점 처리를 빠뜨려요. 1순위가 같을 때의 2순위 정렬을 정하지 않으면 입력 순서에 따라 답이 흔들려요.

한눈에 정리

  • 언제 : 매 순간 최선을 고르는 것이 전체 최적이 될 때예요.
  • 복잡도 : 보통 정렬 O(N log N) 뒤 한 번 순회.
  • 핵심 : 정렬 기준(예를 들어 끝나는 시각)이 풀이의 전부예요.
  • 실수 주의 : 그리디가 항상 옳다고 단정하지 말고 작은 반례로 먼저 확인하세요.

동적 계획법 (DP)

큰 문제를 작은 문제로 쪼개고, 작은 문제의 답을 저장해서 재사용하는 기법이에요. 어려워 보이지만 패턴이 정해져 있어요.

왜 필요한가요? 피보나치로 보기

피보나치를 단순 재귀로 짜면 같은 계산이 수없이 반복돼요. Fib(50)을 단순 재귀로 부르면 작은 값들을 수억 번 다시 계산해요. n이 조금만 더 커지면 호출 수가 폭발해서 사실상 끝나지 않죠. 한 번 구한 답을 배열에 적어두고 재사용하면 O(N)으로 끝나요. 이게 DP의 전부예요.

// 방법 1 : 메모이제이션 (재귀 + 저장)
static long[] memo = new long[100];   // n 최대치+1 크기로 잡아요 (long엔 Fib(92)까지만 담겨요)
static long Fib(int n)
{
    if (n <= 1) return n;
    if (memo[n] != 0) return memo[n];   // 이미 구했으면 재사용 (0이 정답일 수 있는 문제에선 bool 배열로 따로 표시)
    return memo[n] = Fib(n - 1) + Fib(n - 2);
}

// 방법 2 : 테이블 채우기 (반복문, 보통 이쪽을 권해요)
long[] dp = new long[n + 1];
dp[0] = 0; dp[1] = 1;
for (int i = 2; i <= n; i++)
    dp[i] = dp[i - 1] + dp[i - 2];

DP 문제 푸는 3단계

  • 1. 상태 정의 : dp[i]가 무엇을 의미하는지 한국어 문장으로 적어요. "dp[i] = i번째 계단까지 오는 방법의 수"처럼요. 이 정의가 DP의 절반이에요.
  • 2. 점화식 세우기 : dp[i]를 더 작은 dp 값들로 표현해요. "마지막 한 걸음이 1칸이었거나 2칸이었거나" 식으로 마지막 선택을 기준으로 경우를 나누는 게 요령이에요.
  • 3. 초기값 설정 : 가장 작은 문제(dp[0], dp[1])의 답을 직접 채워요. 초기값이 틀리면 전체가 틀려요.

연습 : 계단 오르기

문제 : 한 번에 1칸 또는 2칸을 오를 수 있을 때, N번째 계단까지 오르는 방법의 수는 몇 가지일까요?

상태 : dp[i] = i번째 계단에 도달하는 방법의 수
점화식 : 마지막 걸음이 1칸이었다면 dp[i-1]에서, 2칸이었다면 dp[i-2]에서 왔으니 dp[i] = dp[i-1] + dp[i-2]
초기값 : dp[1] = 1, dp[2] = 2

long[] dp = new long[n + 1];
dp[1] = 1;
if (n >= 2) dp[2] = 2;
for (int i = 3; i <= n; i++)
    dp[i] = dp[i - 1] + dp[i - 2];   // 피보나치와 같은 구조예요

dp 배열이 이렇게 채워져요 : 계단 5칸

i (계단)dp[i]계산
11초기값
22초기값
33dp[2] + dp[1] = 2 + 1
45dp[3] + dp[2] = 3 + 2
58dp[4] + dp[3] = 5 + 3

작은 칸부터 채워 올라가면 5번째 계단까지 오르는 방법은 8가지예요. 각 칸을 딱 한 번씩만 계산하므로 O(N)이에요.

이럴 때 써요 : "방법의 수", "최대/최소 비용", "i번째까지 고려했을 때"라는 표현, 그리고 완전 탐색으로는 경우의 수가 폭발하는 구조예요.

한눈에 정리

  • 언제 : 부분 문제가 겹치고 최적 부분 구조가 있을 때예요.
  • 복잡도 : 상태 수 곱하기 전이 비용.
  • 핵심 : 상태 정의, 점화식, 초기값과 계산 순서를 차례로 정해요.
  • 실수 주의 : 상태나 초기값을 잘못 잡거나 메모이제이션을 빠뜨리면 지수로 폭발해요.

우선순위 큐 (힙)

"지금까지 넣은 것 중 가장 작은(큰) 값"을 O(log N)에 꺼내주는 자료구조예요. 다익스트라, 작업 스케줄링의 심장이에요.

이럴 때 써요

매번 최소값을 찾아 꺼내야 하는데, 그때마다 정렬(O(N log N))이나 전체 순회(O(N))를 하면 느려요. 힙은 넣기와 꺼내기 모두 O(log N)이라 "최소값을 반복해서 꺼내는" 작업에 최적이에요.

C#에서 쓰는 법

// PriorityQueue<요소, 우선순위> : .NET 6 이상에서 제공돼요
// 우선순위가 작을수록 먼저 나와요 (최소 힙)
var pq = new PriorityQueue<string, int>();

pq.Enqueue("세 번째", 3);
pq.Enqueue("첫 번째", 1);
pq.Enqueue("두 번째", 2);

while (pq.Count > 0)
    Console.WriteLine(pq.Dequeue());   // 첫 번째, 두 번째, 세 번째 순으로 나와요

// 최대 힙이 필요하면 우선순위에 부호를 뒤집어 넣는 게 가장 간단해요
maxPq.Enqueue(value, -value);
버전 확인 : PriorityQueue는 .NET 6부터 제공돼요. 응시 플랫폼의 .NET 버전이 낮으면 못 쓸 수 있으니, 시험 환경에서 한 줄 컴파일 테스트를 먼저 해보세요. 안 되면 SortedSet에 (값, 고유번호) 튜플을 넣는 방식으로 대체할 수 있어요.

대표 문제 : 포션 연성 (이 사이트 예시 문제)

"가장 작은 두 값을 꺼내 섞어서 다시 넣기"를 반복하는 문제예요. 매번 최소 두 개가 필요하니 힙이 정답 도구예요.

var pq = new PriorityQueue<int, int>();
foreach (var v in potions) pq.Enqueue(v, v);

int mixCount = 0;
while (pq.Peek() < K)
{
    // 남은 게 하나뿐인데 아직 K 미만이면 실패예요
    if (pq.Count < 2) return -1;

    int a = pq.Dequeue();
    int b = pq.Dequeue();
    int mixed = a + b;
    pq.Enqueue(mixed, mixed);
    mixCount++;
}
복잡도 : 넣기(Enqueue)와 가장 작은 값 빼기(Dequeue)가 각각 O(log N)이에요. "매번 최솟값(또는 최댓값)을 꺼내 처리"를 N번 반복하면 전체 O(N log N)이라, 매번 전체를 훑는 O(N²)보다 훨씬 빨라요.
자주 하는 실수 : ① Enqueue(요소, 우선순위)는 인자가 둘이에요. 우선순위 자리에 엉뚱한 값을 넣어 정렬 기준이 깨지는 실수가 잦아요. ② 최대 힙을 부호 반전으로 흉내 낼 땐 요소는 원래 값, 우선순위만 음수로 넣으세요. ③ 비었을 때 Peek/Dequeue는 예외가 나니 Count를 먼저 확인하세요.

한눈에 정리

  • 언제 : 가장 작은(또는 큰) 값을 반복해서 꺼내야 할 때예요.
  • 복잡도 : 넣기와 꺼내기가 각각 O(log N).
  • 핵심 : C# 는 PriorityQueue(요소, 우선순위), 최대 힙은 우선순위 부호를 뒤집어요.
  • 실수 주의 : 빈 큐에서 Peek 또는 Dequeue, 우선순위 인자 혼동을 조심하세요.

투 포인터와 누적 합

O(N²)짜리 구간 문제를 O(N)으로 떨어뜨리는 두 가지 기술이에요. "연속 부분 배열"이라는 말이 보이면 이 둘을 떠올리세요.

누적 합 : 구간 합을 O(1)로

구간 합을 매번 더하면 질의마다 O(N)이에요. 미리 "처음부터 i까지의 합"을 배열로 만들어 두면, 어떤 구간 합이든 뺄셈 한 번으로 구해져요.

// prefix[i] = arr[0]부터 arr[i-1]까지의 합
long[] prefix = new long[n + 1];
for (int i = 0; i < n; i++)
    prefix[i + 1] = prefix[i] + arr[i];

// 구간 [l, r]의 합 (0-based, 양 끝 포함)
long rangeSum = prefix[r + 1] - prefix[l];
이럴 때 써요 : "구간 합을 여러 번 묻는다", "M개의 질의가 주어진다". 질의가 많을수록 누적 합의 이득이 커져요. 합이 int 범위를 넘기 쉬우니 long을 쓰세요.

투 포인터 : 양쪽에서 좁혀가기

정렬된 배열에서 "합이 target인 두 수 찾기"를 예로 들면, 양 끝에 포인터를 두고 합이 크면 오른쪽을 줄이고 작으면 왼쪽을 늘려요. 각 포인터가 최대 N번만 움직이니 O(N)이에요.

// 정렬된 배열에서 합이 target인 쌍이 있는지 확인해요
int left = 0, right = arr.Length - 1;
while (left < right)
{
    long sum = arr[left] + arr[right];
    if (sum == target) return true;
    else if (sum < target) left++;    // 합을 키워야 하니 왼쪽 전진
    else right--;                      // 합을 줄여야 하니 오른쪽 후퇴
}
return false;

슬라이딩 윈도우 : 같은 방향으로 미는 투 포인터

"합이 S 이상인 가장 짧은 연속 부분 배열" 같은 문제는 두 포인터를 같은 방향으로 밀어요. 창문(구간)을 오른쪽으로 넓혔다가 조건을 만족하면 왼쪽을 좁히는 식이에요.

int start = 0, best = int.MaxValue;
long windowSum = 0;

for (int end = 0; end < n; end++)
{
    windowSum += arr[end];               // 창을 오른쪽으로 한 칸 넓혀요

    // 조건을 만족하는 동안 왼쪽을 최대한 좁혀 최소 길이를 갱신해요
    while (windowSum >= S)
    {
        best = Math.Min(best, end - start + 1);
        windowSum -= arr[start];
        start++;
    }
}

두 포인터 따라가기 : 정렬된 [1, 3, 4, 6, 8, 11], target = 10

0, 51 + 11 = 12크다 → right 한 칸 당김
0, 41 + 8 = 9작다 → left 한 칸 밈
1, 43 + 8 = 11크다 → right 한 칸 당김

두 포인터가 서로를 향해 한 칸씩만 움직이니 전체를 O(N)에 끝내요. 2중 반복문 O(N²)을 한 줄짜리 규칙으로 대체하는 셈이죠.

자주 하는 실수 : ① 양 끝 투 포인터는 정렬된 배열에서만 "크면 줄이고 작으면 늘리기"가 성립해요. 정렬 안 됐으면 먼저 정렬하세요. ② 누적 합에서 구간 [l, r] 합은 prefix[r+1] - prefix[l]예요. prefix 배열 크기를 N+1로 잡고 인덱스 한 칸 차이(off-by-one)를 늘 확인하세요.

한눈에 정리

  • 언제 : 연속 부분 배열이나 구간 합 질의가 보이면 떠올려요.
  • 복잡도 : O(N), 누적 합을 전처리하면 질의당 O(1).
  • 핵심 : 양 끝(정렬된 배열) 또는 같은 방향(슬라이딩 윈도우), 구간 합은 prefix[r+1] - prefix[l].
  • 실수 주의 : 양 끝 투 포인터는 정렬이 전제예요. 인덱스 한 칸 차이와 long 을 확인하세요.

최단 경로 (다익스트라)

간선마다 가중치(거리, 비용)가 있는 그래프에서 한 점부터 모든 점까지의 최단 거리를 구하는 알고리즘이에요. BFS가 안 통하는 "가중치 있는 최단 거리"의 정답 도구예요.

왜 BFS로는 안 되나요?

BFS의 "처음 도달 = 최단"은 모든 간선의 비용이 같을 때만 성립해요. 간선마다 비용이 다르면, 칸 수는 적어도 비용 합이 더 큰 경로가 먼저 도달할 수 있어요. 그래서 "지금까지 거리가 가장 짧은 정점"을 항상 먼저 처리해야 하고, 그 일을 우선순위 큐가 맡아요. 11번 주제(우선순위 큐)가 여기서 쓰여요.

핵심 구현 (우선순위 큐 방식)

// graph[u] = (도착 정점 v, 가중치 w) 목록, 인접 리스트예요
var graph = new List<(int v, int w)>[n + 1];
for (int i = 1; i <= n; i++) graph[i] = new();

long[] dist = new long[n + 1];
Array.Fill(dist, long.MaxValue);   // 처음엔 모두 무한대로 둬요

var pq = new PriorityQueue<int, long>();  // (정점, 그 정점까지 거리)
dist[start] = 0;
pq.Enqueue(start, 0);

while (pq.Count > 0)
{
    pq.TryDequeue(out int u, out long d);
    // 이미 더 짧게 확정된 정점이면 건너뛰어요. 중복으로 넣어도 이 검사가 걸러줘서 decrease-key가 필요 없어요
    if (d > dist[u]) continue;

    foreach (var (v, w) in graph[u])
    {
        // u를 거쳐 가는 게 더 짧으면 거리를 갱신해요 (이완, relaxation)
        if (dist[u] + w < dist[v])
        {
            dist[v] = dist[u] + w;
            pq.Enqueue(v, dist[v]);
        }
    }
}
주의 두 가지 : ① 다익스트라는 음수 간선이 있으면 틀려요 (그땐 벨만-포드를 써요). ② 거리 합이 int 범위를 쉽게 넘으니 dist는 long으로 두세요.
복잡도 : 정점 V개, 간선 E개일 때 O(E log V)예요. "가중치 있는 그래프 + 최단/최소 비용"이 보이면 다익스트라를 1순위로 떠올리세요.

거리(dist)가 이렇게 줄어들어요

간선 1→2(2), 1→3(5), 2→3(1), 2→4(7), 3→4(3). 시작 정점은 1이고, INF는 아직 도달 못 한 상태예요.

항상 "현재까지 거리가 가장 짧은 정점"을 우선순위 큐로 먼저 꺼내 처리하는 게 핵심이에요. 1→2→3→4(2+1+3=6)가 1→3→4(5+3=8)보다 짧다는 걸 자동으로 찾아냈죠.

  • 이럴 때 써요
  • 간선마다 가중치(비용, 시간, 거리)가 다른 그래프에서 최소 비용 경로를 구할 때
  • 가중치가 모두 같다면 다익스트라 대신 더 간단한 BFS로 충분해요
  • 음수 간선이 있으면 다익스트라는 못 쓰고 벨만-포드를 써요

한눈에 정리

  • 언제 : 음수 간선이 없는 가중 그래프의 최단 거리예요.
  • 복잡도 : O(E log V).
  • 핵심 : 우선순위 큐로 가장 가까운 정점부터 확정하고, 거리는 long 으로 둬요.
  • 실수 주의 : 이미 더 짧게 확정된 정점은 건너뛰고, 음수 간선에는 쓸 수 없어요.

분리 집합 (Union-Find)

"이 둘이 같은 그룹에 속하나?"를 거의 O(1)에 답하는 자료구조예요. 네트워크 연결, 사이클 판별, 그룹 묶기 문제의 핵심이에요.

두 가지 연산이 전부예요

  • Find(x) : x가 속한 그룹의 "대표(루트)"를 찾아요.
  • Union(a, b) : a가 속한 그룹과 b가 속한 그룹을 하나로 합쳐요.

"두 원소가 같은 그룹인가?"는 Find(a) == Find(b) 한 줄로 판별돼요. 각 원소는 자기 부모를 가리키는 배열 하나로 트리처럼 표현해요.

static int[] parent;

// 초기화 : 처음엔 모두 자기 자신이 대표예요
parent = new int[n + 1];
for (int i = 1; i <= n; i++) parent[i] = i;

static int Find(int x)
{
    if (parent[x] == x) return x;
    // 경로 압축 : 찾은 루트를 곧장 부모로 바꿔 다음 탐색을 빠르게 해요
    return parent[x] = Find(parent[x]);
}

static void Union(int a, int b)
{
    int ra = Find(a), rb = Find(b);
    if (ra != rb) parent[ra] = rb;   // 한쪽 루트를 다른 쪽에 붙여요
}
경로 압축이 핵심이에요 : parent[x] = Find(...)로 찾은 루트를 바로 부모로 갱신하면, 트리가 점점 납작해져 이후 Find가 거의 O(1)이 돼요. 이 한 줄을 빼면 한 줄로 길게 이어진 트리에서 O(N)으로 느려질 수 있어요.

parent 배열이 이렇게 변해요 (원소 1~5)

이제 1, 2, 3, 4 는 대표가 4인 한 그룹이고 5만 따로예요. Find(1) == Find(4) 이므로 같은 그룹이죠. 경로 압축 덕에 트리가 납작해져 다음 Find 는 거의 한 번에 끝나요.

복잡도 : 경로 압축을 적용하면 Find/Union 한 번이 거의 O(1)이에요(정확히는 매우 느리게 자라는 역아커만 함수). 그래서 "같은 그룹인가" 질의가 수십만 번 와도 빠르게 답해요.

대표 활용 : 사이클 판별

간선 (a, b)를 하나씩 추가할 때, Find(a) == Find(b)인데 또 잇는다면 그 간선이 사이클을 만든다는 뜻이에요. 최소 신장 트리(크루스칼)에서 사이클을 거르는 데 그대로 쓰여요.

  • 이럴 때 써요
  • "두 노드가 연결돼 있나/같은 그룹인가" 질의가 많을 때
  • "그룹(연결 요소)이 몇 개인가" 셀 때
  • 간선을 더하며 사이클이 생기는지 판별할 때 (크루스칼 MST)

한눈에 정리

  • 언제 : 연결 여부, 연결 요소 개수, 사이클 판정에 써요.
  • 복잡도 : 경로 압축을 쓰면 사실상 O(1) 에 가까워요.
  • 핵심 : Find(경로 압축) 와 Union 으로 루트를 비교해요.
  • 실수 주의 : 경로 압축을 빼면 느려지고, 합칠 때는 루트끼리 이어야 해요.

비트마스킹

정수 하나의 각 비트를 "켜짐/꺼짐" 스위치로 보고, 여러 개의 참/거짓 상태를 정수 하나에 압축하는 기법이에요. 부분집합 탐색과 상태 압축 DP의 토대예요.

기본 비트 연산 네 가지

int state = 0;   // 0b0000, 아무것도 안 켜진 상태

state |= (1 << i);              // i번째 비트 켜기 (포함시키기)
state &= ~(1 << i);             // i번째 비트 끄기 (제외하기)
state ^= (1 << i);              // i번째 비트 토글 (반전)
bool on = (state & (1 << i)) != 0;  // i번째가 켜져 있는지 확인

핵심은 1 << i예요. 1을 왼쪽으로 i칸 밀어 "i번째 비트만 켜진 수"를 만든 뒤, |(켜기) &(확인) & ~(끄기)와 조합해요.

모든 부분집합 순회

원소가 N개일 때 부분집합은 2N개예요. 0부터 2N-1까지의 정수가 각각 하나의 부분집합에 대응해요. N이 20 이하일 때 쓰는 정석 패턴이에요.

for (int s = 0; s < (1 << n); s++)   // 2^n개의 모든 상태
{
    for (int i = 0; i < n; i++)
        if ((s & (1 << i)) != 0)    // i번째 원소가 이 부분집합에 포함됐다면
            Use(arr[i]);
}

// 켜진 비트(원소) 개수는 이렇게 한 번에 셀 수 있어요
int cnt = System.Numerics.BitOperations.PopCount((uint)s);
주의 : 1 << i는 int 연산이라 i가 31 이상이면 넘쳐요. 원소가 32개를 넘으면 1L << i처럼 long으로 미세요. 그리고 우선순위 때문에 (state & (1 << i)) != 0의 괄호를 꼭 챙기세요. 마스크가 long이면 켜진 비트 수도 PopCount((ulong)s) 로 세요.

정수 하나가 부분집합 하나 : 원소 [A, B, C]

비트 0은 A, 비트 1은 B, 비트 2는 C예요. s를 0부터 7까지 돌리면 모든 부분집합이 정확히 한 번씩 나와요.

s (2진수)포함되는 원소
0 (000){ } 공집합
1 (001){ A }
2 (010){ B }
3 (011){ A, B }
4 (100){ C }
5 (101){ A, C }
6 (110){ B, C }

복잡도 : 모든 부분집합을 도는 건 O(2N x N)이에요. 그래서 N이 20 안팎일 때만 써요(220이 약 100만). N이 더 크면 다른 방법을 찾아야 해요.
  • 이럴 때 써요
  • N이 작을 때(20 이하) "모든 부분집합 또는 조합을 다 살펴보라"는 문제
  • 여러 개의 켜짐/꺼짐 상태를 정수 하나로 묶어 DP 상태로 쓸 때(상태 압축 DP)
  • 방문한 정점 집합처럼 "집합"을 빠르게 비교하고 합쳐야 할 때

한눈에 정리

  • 언제 : 원소가 20개 이하인 작은 집합의 상태를 정수 하나로 표현할 때예요.
  • 복잡도 : 모든 부분집합을 보면 2^N.
  • 핵심 : i번째 비트는 (mask >> i) & 1, 켜기는 mask | (1 << i).
  • 실수 주의 : 연산자 우선순위 괄호를 빠뜨리기 쉽고, 1 << i 가 31 이상이면 1L 을 쓰세요.

구현과 시뮬레이션

특별한 알고리즘 없이 문제에 적힌 규칙을 그대로 코드로 옮기는 유형이에요. 난이도는 알고리즘이 아니라 "꼼꼼함"에서 갈려요. 출제 비중이 가장 높은 축에 들어요.

왜 따로 다루나요?

구현 문제는 "어떻게 풀지"는 명확한데 "정확히 옮기기"가 어려워요. 조건이 여러 개 겹치고, 좌표, 방향, 회전이 얽히면 작은 실수 하나로 통째로 틀려요. 그래서 규칙을 코드로 옮기기 전에 글로 단계를 쪼개는 습관이 점수를 좌우해요.

방향 전환 : 회전을 인덱스 산수로

"우회전", "좌회전"이 자주 나와요. 방향 배열을 시계 방향으로 정렬해 두면 회전이 +1, -1 인덱스 계산으로 끝나요.

// 북, 동, 남, 서를 시계 방향 순서로 둬요
int[] dx = { -1, 0, 1, 0 };
int[] dy = { 0, 1, 0, -1 };
int dir = 0;   // 현재 북쪽을 보고 있어요

dir = (dir + 1) % 4;       // 우회전 (시계 방향 한 칸)
dir = (dir + 3) % 4;       // 좌회전 (-1 대신 +3을 더해 음수를 피해요)
음수 나머지 함정 : C#에서 (-1) % 4는 0이 아니라 -1이에요. 좌회전을 (dir - 1) % 4로 쓰면 음수 인덱스가 나와 예외가 터져요. "한 칸 뒤로 = 전체 크기 빼기 1만큼 앞으로"로 바꿔 (dir + 3) % 4로 쓰세요.

구현 문제를 안정적으로 푸는 순서

  • 1. 규칙을 번호 매긴 글머리표로 적어요. ("① 앞 칸이 벽이 아니면 전진 ② 벽이면 우회전 ...")
  • 2. 상태를 변수로 정해요. (현재 좌표 x, y / 방향 dir / 누적 점수 등)
  • 3. 한 글머리표 = 한 코드 블록으로 옮기고, 그때마다 예시로 검증해요.
  • 4. 격자 밖 검사(nx < 0 || nx >= n ...)와 종료 조건을 가장 먼저 챙겨요.

로봇 시뮬레이션 따라가기

3 x 3 격자, 시작 (0,0)에서 동쪽을 보고 있어요. 규칙은 "앞 칸이 격자 안이면 전진, 격자 밖이면 우회전"이에요.

단계위치 (행, 열)방향앞 칸행동
시작(0, 0)동(0, 1) 안전진
1(0, 1)동(0, 2) 안전진
2(0, 2)동(0, 3) 밖우회전 → 남
3(0, 2)남(1, 2) 안전진
4(1, 2)남(2, 2) 안전진

방향(dir)과 좌표(행, 열)만 상태로 들고, 규칙을 한 줄씩 그대로 옮기면 돼요. 머릿속으로 한 번 굴려본 표가 곧 코드의 명세서예요.

복잡도 : 보통 한 칸과 한 턴을 상수 시간에 처리하므로 전체는 O(격자 칸 수) 또는 O(시뮬레이션 단계 수)예요. 알고리즘이 아니라 단계 수가 그대로 시간이에요.

한눈에 정리

  • 언제 : 정해진 규칙을 정확히 따라가야 하는 문제예요.
  • 복잡도 : 시뮬레이션 단계 수에 비례해요.
  • 핵심 : 방향 배열(dx, dy)과 작은 보조 함수로 단계를 나눠요.
  • 실수 주의 : 경계와 회전, 인덱스 실수가 잦으니 작은 예시로 손으로 따라가 보세요.

길찾기 알고리즘 (A*와 미로 탐색)

시작점에서 목표점까지 최단 경로를 찾는 문제예요. 게임 길찾기, 미로 탈출, 지도 내비게이션의 핵심이고, 상황에 맞게 BFS, 다익스트라, A* 중 알맞은 것을 고르는 눈이 중요해요.

먼저, 셋 중 무엇을 쓸까요?

상황알맞은 알고리즘이유
모든 이동 비용이 같음 (한 칸 = 1)BFS (8번 주제)가까운 곳부터 퍼져, 처음 도달한 순간이 곧 최단이에요
이동 비용이 칸/간선마다 다름다익스트라 (13번 주제)지금까지 거리가 가장 짧은 정점부터 확정해요
목표가 한 점으로 정해져 있고 빠른 길찾기가 필요A*목표 방향으로 탐색을 집중해, 같은 최단을 더 적게 탐색해요
셋은 한 가족이에요 : A*는 다익스트라에 "목표까지 남은 거리 추정값(휴리스틱)"을 더한 거예요. 휴리스틱을 0으로 두면 A*는 그대로 다익스트라가 되고, 모든 비용이 1인 다익스트라는 사실상 BFS예요. 그래서 하나를 이해하면 나머지가 쉬워져요.

A*의 아이디어 : f = g + h

A*는 다음에 살펴볼 칸을 고를 때 두 값을 더해 판단해요.

  • g(n) : 시작점부터 지금 칸까지 실제로 든 비용이에요 (다익스트라의 거리와 같아요).
  • h(n) : 지금 칸부터 목표까지 예상 비용(휴리스틱)이에요. 벽을 무시하고 어림한 거리예요.
  • f(n) = g(n) + h(n) : 이 칸을 거치는 전체 경로의 예상 길이예요. 우선순위 큐에서 f가 가장 작은 칸부터 꺼내요.

상하좌우만 움직이는 그리드에서는 맨해튼 거리(가로 차이 + 세로 차이)를 휴리스틱으로 써요. 대각선 이동이 가능하면 체비쇼프 거리나 유클리드 거리를 쓰고요.

최단을 보장하는 조건 : 휴리스틱 h가 실제 남은 비용을 절대 과대평가하지 않아야 해요(이를 "허용 가능한 휴리스틱"이라 불러요). 맨해튼 거리는 벽이 있어도 실제 거리보다 클 수 없으니 안전해요. 과대평가하는 h를 쓰면 빨라지는 대신 최단이 아닐 수 있어요.

그리드 미로에서의 A*

// grid : 0 = 길, 1 = 벽. (0,0)에서 (n-1, m-1)까지 최소 칸 수를 구해요
static int AStar(int[,] grid, int n, int m)
{
    int[] dx = { -1, 1, 0, 0 };
    int[] dy = { 0, 0, -1, 1 };

    // 휴리스틱 : 목표까지 맨해튼 거리 (상하좌우만 이동하므로 과대평가가 없어요)
    int H(int x, int y) => Math.Abs(x - (n - 1)) + Math.Abs(y - (m - 1));

    var g = new int[n, m];   // g[x,y] = 시작부터 실제 비용
    for (int i = 0; i < n; i++)
        for (int j = 0; j < m; j++) g[i, j] = int.MaxValue;

    var pq = new PriorityQueue<(int x, int y), int>();  // 우선순위 = f = g + h
    g[0, 0] = 0;
    pq.Enqueue((0, 0), H(0, 0));

    while (pq.Count > 0)
    {
        var (x, y) = pq.Dequeue();
        // 목표를 처음 꺼낸 순간이 최단이에요 (허용 가능한 휴리스틱 덕분)
        if (x == n - 1 && y == m - 1) return g[x, y] + 1;  // 칸 수로 환산

        for (int d = 0; d < 4; d++)
        {
            int nx = x + dx[d], ny = y + dy[d];
            if (nx < 0 || nx >= n || ny < 0 || ny >= m) continue;  // 격자 밖
            if (grid[nx, ny] == 1) continue;                          // 벽

            int ng = g[x, y] + 1;   // 한 칸 이동 비용 1
            if (ng < g[nx, ny])
            {
                g[nx, ny] = ng;
                pq.Enqueue((nx, ny), ng + H(nx, ny));   // f = g + h 로 우선순위 부여
            }
        }
    }
    return -1;   // 도달 불가
}
다익스트라와 거의 같죠? : 13번 주제의 다익스트라와 비교해 보세요. 핵심 차이는 한 군데예요. 우선순위에 ng(=g) 대신 ng + H(...)(=g+h)를 넣어요. 이 한 줄이 탐색을 목표 쪽으로 끌어당겨요. (다익스트라에 있던 "이미 더 짧게 확정됐으면 건너뛰기" 가드는 여기선 생략했는데, 정답은 같고 넣으면 조금 더 빨라져요.)

미로 문제에서 빠지기 쉬운 곳

  • 경계 밖 검사(nx, ny가 격자 안인지)를 가장 먼저 해요. 안 하면 인덱스 예외가 나요.
  • 벽(이동 불가 칸)과 방문 여부를 둘 다 확인해요.
  • "칸 수"를 세는지 "이동 횟수"를 세는지 정확히 읽어요. 시작 칸 포함 여부로 답이 1 차이 나요.
  • 가중치가 없는 단순 미로라면 굳이 A*를 쓰지 말고 BFS를 쓰세요. A*는 코드가 길어 시험장에서 실수 여지가 커요.
복잡도 : 최악에는 다익스트라와 같은 O(E log V)예요. 다만 좋은 휴리스틱이 탐색을 목표 쪽으로 끌어당겨, 실제로 꺼내 보는 칸 수가 크게 줄어드는 게 이득이에요. 휴리스틱이 실제 거리를 넘기지 않으면(허용 가능) 최단을 보장해요.

한눈에 정리

  • 언제 : 칸마다 드는 비용이 다른 격자의 최단 비용이에요.
  • 복잡도 : 다익스트라류로 O(칸 수 곱하기 log).
  • 핵심 : 비용이 1 이 아니면 BFS 가 아니라 다익스트라 또는 A*(f = g + h)를 써요.
  • 실수 주의 : 비용이 다른데 BFS 를 쓰거나, 방문 확정 시점을 잘못 잡지 마세요.
QUIZ 1A*에서 휴리스틱 h를 항상 0으로 두면 어떤 알고리즘과 똑같이 동작할까요?
정답: 다익스트라
f = g + h에서 h가 0이면 f = g가 되어, "지금까지 실제 비용이 가장 작은 칸"부터 꺼내는 다익스트라와 완전히 같아져요. A*는 다익스트라에 목표 방향 힌트(h)를 얹은 확장이라고 이해하면 돼요. h가 좋을수록 불필요한 탐색이 줄어요.

위상 정렬 (Topological Sort)

"A를 끝내야 B를 할 수 있다" 같은 선후 관계가 있는 작업들을 가능한 순서로 줄 세우는 방법이에요. 선수 과목, 빌드 순서, 작업 일정 문제에 나와요.

이럴 때 써요

  • 방향이 있는 간선(a → b = a를 b보다 먼저)으로 표현되는 의존 관계가 있을 때 써요.
  • 사이클이 없는 방향 그래프(DAG)에서만 가능해요. 서로 물고 물리면 순서를 정할 수 없어요.

진입 차수 + 큐 (카안 알고리즘)

각 정점으로 들어오는 간선 수를 진입 차수(in-degree)라고 해요. 진입 차수가 0인 정점은 "먼저 해도 되는 일"이에요. 그걸 꺼내 처리하고, 그 정점이 가리키던 곳들의 진입 차수를 1씩 줄이는 걸 반복해요.

// graph[u] = u 다음에 올 수 있는 정점 목록, indeg[v] = v로 들어오는 간선 수
var queue = new Queue<int>();
for (int i = 1; i <= n; i++)
    if (indeg[i] == 0) queue.Enqueue(i);   // 시작점 : 선행 작업이 없는 정점

var order = new List<int>();
while (queue.Count > 0)
{
    int u = queue.Dequeue();
    order.Add(u);
    foreach (int v in graph[u])
        if (--indeg[v] == 0) queue.Enqueue(v);   // 선행이 모두 끝났으면 큐에 넣어요
}
사이클 판별 : 결과 order에 담긴 정점 수가 N보다 적으면, 남은 정점들이 서로 사이클을 이뤄 진입 차수가 끝내 0이 되지 못한 거예요. "순서를 정할 수 없다"는 답은 order.Count < n으로 잡아내요.

동작 따라가기 : 1→2, 1→3, 2→4, 3→4, 4→5

처음 진입 차수는 [1:0, 2:1, 3:1, 4:2, 5:1] 이에요. 진입 차수 0인 1만 큐에서 시작해요.

꺼낸 정점줄어든 진입 차수큐 상태order
12 → 0, 3 → 0[2, 3]1
24 → 1[3]1 2
34 → 0[4]1 2 3
45 → 0[5]1 2 3 4
5없음[ ]1 2 3 4 5

진입 차수가 0이 된 정점을 큐에 넣어 순서대로 꺼내면, 선후 관계를 어기지 않는 줄 세우기 1 2 3 4 5가 나와요. 5개 다 담겼으니 사이클도 없어요.

복잡도 : 정점 V개, 간선 E개를 한 번씩만 보므로 O(V + E)예요. 각 정점은 큐에 한 번, 각 간선은 진입 차수를 줄일 때 한 번 쓰여요.

한눈에 정리

  • 언제 : 선후 관계가 있는 작업의 순서를 정할 때예요.
  • 복잡도 : O(V + E).
  • 핵심 : 진입 차수 0 인 것부터 처리하고, 사전순이 필요하면 우선순위 큐를 써요.
  • 실수 주의 : 끝까지 다 빼지 못하면 순환이라 -1, 진입 차수 갱신을 빠뜨리지 마세요.

수학

소수와 에라토스테네스의 체

소수(1보다 큰 수 중 1과 자기 자신만 약수로 갖는 수, 1은 소수가 아니에요) 판정은 단골이에요. 하나만 판정할 땐 √N까지만 나눠보고, 범위 전체를 구할 땐 "체"로 한 번에 걸러요.

하나만 판정 : √N까지만 나눠봐요

static bool IsPrime(int x)
{
    if (x < 2) return false;
    for (int i = 2; (long)i * i <= x; i++)   // i*i가 x를 넘기 전까지만
        if (x % i == 0) return false;          // 약수를 찾으면 소수가 아니에요
    return true;
}

약수는 √N을 기준으로 쌍을 이루기 때문에, √N까지만 확인하면 충분해요. i * i <= x 조건이 그 √N 경계예요. (i*i가 int 범위를 넘을 수 있어 long으로 비교해요.)

범위 전체 : 에라토스테네스의 체

2부터 시작해 각 소수의 배수를 모두 지워나가면, 끝까지 안 지워진 수가 소수예요. N 이하 모든 소수를 O(N log log N)에 구해요.

bool[] notPrime = new bool[n + 1];   // notPrime[k] = k가 소수가 아니면 true
for (int i = 2; (long)i * i <= n; i++)
{
    if (notPrime[i]) continue;            // 이미 지워졌으면 그 배수도 이미 지워졌어요
    for (int j = i * i; j <= n; j += i)   // i*i부터 시작 (그 앞 배수는 이미 지워짐)
        notPrime[j] = true;
}
// k >= 2 이고 notPrime[k] == false 이면 소수예요
두 가지 최적화 : ① 배수 지우기를 i*i부터 시작해요 (2i, 3i처럼 더 작은 소수의 배수는 이미 지워졌어요). ② 바깥 루프는 i*i <= n까지만 돌아도 충분해요. 이 둘이 체를 빠르게 만들어요.

체가 이렇게 걸러내요 : n = 20

단계지우는 수
i = 24, 6, 8, 10, 12, 14, 16, 18, 20
i = 39, 15 (6, 12, 18은 이미 지워짐)
i = 4이미 지워진 수(합성수)라 배수 지우기는 건너뛰어요
i = 55*5 = 25 > 20 이라 바깥 루프(i*i ≤ n)가 여기서 끝나요
결과안 지워진 수 = 2, 3, 5, 7, 11, 13, 17, 19 (20 이하 소수)

각 소수의 배수를 한 번씩만 훑어 지우기 때문에, 하나하나 √N 판정하는 것보다 범위 전체를 훨씬 빠르게 구해요.

  • 이럴 때 써요
  • "N 이하의 모든 소수", "소수의 개수", "범위 안 소수를 여러 번 조회" 같은 표현 → 에라토스테네스의 체
  • "이 수 하나가 소수인가"만 묻고 수가 클 때 → √N 판정
  • 소인수분해나 약수 개수도 체를 변형해 빠르게 구할 수 있어요

한눈에 정리

  • 언제 : 일정 범위 안의 소수를 여러 개 판정할 때예요.
  • 복잡도 : 체는 O(N log log N), 한 수 판정은 O(루트 N).
  • 핵심 : 2부터 배수를 지우고, 지우기는 i*i 부터 시작해요.
  • 실수 주의 : 0 과 1 처리를 빠뜨리기 쉽고, 단일 판정은 루트 N 까지만 보면 돼요.
QUIZ 1어떤 수 x가 소수인지 하나만 판정할 때, 나눗셈은 어디까지만 해보면 충분할까요?

약수는 √x를 기준으로 쌍을 이뤄요. 예를 들어 36 = 4 x 9에서 4가 약수면 9도 약수죠. 그래서 √x까지만 확인하면 더 큰 짝꿍 약수는 자동으로 걸러져요. 코드에선 i * i <= x로 표현해요.

유클리드 호제법 (최대공약수)

두 수의 최대공약수(GCD)를 아주 빠르게 구하는 방법이에요. 최소공배수(LCM)도 여기서 바로 나와요.

핵심 한 줄 : GCD(a, b) = GCD(b, a % b)

큰 수를 작은 수로 나눈 나머지로 계속 바꿔가다 보면 나머지가 0이 되는 순간이 와요. 그때 나누던 수가 최대공약수예요.

static long Gcd(long a, long b)
{
    while (b != 0)
    {
        long t = a % b;
        a = b;
        b = t;
    }
    return a;          // b가 0이 된 순간의 a가 최대공약수예요
}

// 최소공배수 = 두 수의 곱 / 최대공약수
static long Lcm(long a, long b) => a / Gcd(a, b) * b;   // 곱하기 전에 나눠 오버플로를 피해요
오버플로 함정 : LCM을 a * b / gcd로 쓰면 a * b가 먼저 계산되며 long 범위를 넘길 수 있어요. a / gcd * b처럼 먼저 나누고 곱하면 중간 값이 작아져 안전해요. (a는 gcd로 항상 나누어떨어지니 정확해요.)

동작 과정 따라가기 : GCD(48, 18)

큰 수를 작은 수로 나눈 나머지로 계속 바꿔요. 한 단계마다 (a, b) 가 (b, a % b) 로 옮겨가요.

단계aba % b
시작481812
1회18126
2회1260
끝60b가 0이라 멈춤

b가 0이 된 순간의 a, 즉 6이 48과 18의 최대공약수예요. 나머지로 바꿀 때마다 값이 빠르게 줄어드는 게 보이죠.

복잡도 : O(log(min(a, b)))예요. 나머지 연산을 거칠 때마다 값이 빠르게 작아져, 두 수가 아무리 커도 수십 번 안에 끝나요.
  • 이럴 때 써요
  • "최대공약수", "기약분수로 줄여라", "두 수가 서로소인지 판별" 같은 표현이 보일 때
  • 최소공배수가 필요할 때 (LCM = a / GCD x b)
  • 분수 계산에서 분자와 분모를 약분해야 할 때

한눈에 정리

  • 언제 : 두 수의 최대공약수와 최소공배수가 필요할 때예요.
  • 복잡도 : O(log min(a, b)).
  • 핵심 : gcd(a, b) = gcd(b, a % b), lcm 은 a / gcd 에 b 를 곱해 구해요.
  • 실수 주의 : lcm 은 커지기 쉬우니 a / gcd 를 먼저 하고 long 을 쓰세요.
QUIZ 1유클리드 호제법에서 최대공약수가 확정되는 순간은 언제일까요?

자료구조

트리 기본과 순회

사이클 없이 위에서 아래로 뻗어 내려가는 자료구조예요. 폴더 구조, 가계도, 댓글의 답글이 모두 트리이고 코딩 테스트에 아주 자주 나와요.

트리란 무엇인가

노드 N개를 간선 N-1개로 이어, 어떤 두 노드 사이 경로가 딱 하나뿐인 그래프예요. 맨 위 노드를 루트, 자식이 없는 끝 노드를 리프라고 불러요. "사이클이 없는 연결 그래프"라는 점이 일반 그래프와 다른 핵심이에요.

그래프의 특수한 경우예요 : 트리는 사이클이 없어서 DFS/BFS가 그대로 통해요. 방문 배열 대신 "나를 부른 부모만 빼고 자식으로 내려가면" 돼서 더 간단해요.

이진 트리 표현

자식이 최대 둘인 트리를 이진 트리라고 해요. 노드마다 왼쪽과 오른쪽 자식을 가리키면 돼요.

class Node
{
    public int val;
    public Node left, right;
}

세 가지 순회 : 전위 / 중위 / 후위

나 자신을 언제 방문(출력)하느냐의 차이예요. 전위는 "나 먼저", 중위는 "왼쪽 다음 나", 후위는 "두 자식을 다 본 뒤 나"예요.

void Preorder(Node n)   // 전위: 나 -> 왼 -> 오
{
    if (n == null) return;
    Visit(n.val);
    Preorder(n.left);
    Preorder(n.right);
}

void Inorder(Node n)    // 중위: 왼 -> 나 -> 오 (탐색 트리면 오름차순)
{
    if (n == null) return;
    Inorder(n.left);
    Visit(n.val);
    Inorder(n.right);
}

void Postorder(Node n)  // 후위: 왼 -> 오 -> 나 (자식부터 정리)
{
    if (n == null) return;
    Postorder(n.left);
    Postorder(n.right);
    Visit(n.val);
}

아래 트리를 세 방식으로 순회하면 방문 순서가 이렇게 달라져요.

        1
       / \
      2   3
     / \
    4   5
순회방문 순서
전위1 2 4 5 3
중위4 2 5 1 3
후위4 5 2 3 1
레벨 순회는 BFS예요 : 위에서 아래로 같은 깊이를 왼쪽부터 보고 싶으면 큐를 써요. 위 트리는 1 2 3 4 5 순서가 돼요.
  • 이럴 때 써요
  • 부모와 자식으로 자료가 계층을 이룰 때 (폴더, 조직도, 카테고리)
  • 이진 탐색 트리에서 정렬된 순서가 필요할 때 (중위 순회)
  • 자식의 결과를 모아 부모를 계산할 때 (후위 순회, 트리 DP의 토대)

한눈에 정리

  • 언제 : 사이클 없는 계층 구조를 다룰 때예요.
  • 복잡도 : 모든 순회가 노드 수에 비례하는 O(N).
  • 핵심 : 나를 언제 방문하느냐로 전위/중위/후위가 갈려요.
  • 실수 주의 : 무방향 간선으로 트리를 만들면, 내려갈 때 부모로 되돌아가지 않게 부모를 빼고 재귀하세요.
QUIZ 1이진 탐색 트리를 오름차순으로 출력하려면 어떤 순회를 써야 할까요?
정답: 중위 순회
이진 탐색 트리는 왼쪽 자식이 나보다 작고 오른쪽 자식이 크다는 규칙이 있어요. 그래서 왼쪽 -> 나 -> 오른쪽 순으로 방문하는 중위 순회가 값을 오름차순으로 뽑아 줘요.
QUIZ 2노드가 N개인 트리의 간선 수는 몇 개일까요?
정답: N - 1개
트리는 루트를 뺀 모든 노드가 부모와 잇는 간선을 정확히 하나씩 가져요. 그래서 간선은 항상 노드보다 하나 적은 N-1개이고, 이보다 많으면 사이클이 생겨 트리가 아니에요.
동적 계획법

DP 핵심 유형 (LIS / LCS / 배낭)

동적 계획법은 점화식이 전부예요. 시험에 반복해서 나오는 세 가지 대표 유형의 점화식을 손에 익히면 변형 문제도 풀려요.

LIS : 최장 증가 부분 수열

수열에서 순서를 지키며 고른, 점점 커지는 가장 긴 부분 수열의 길이예요. dp[i] 를 "i번째로 끝나는 증가 수열의 최대 길이"로 두면 돼요.

int[] dp = new int[n];
int best = 0;
for (int i = 0; i < n; i++)
{
    dp[i] = 1;                          // 최소한 자기 자신
    for (int j = 0; j < i; j++)
        if (a[j] < a[i])
            dp[i] = Math.Max(dp[i], dp[j] + 1);
    best = Math.Max(best, dp[i]);
}
더 빠른 방법 : 위는 O(N^2)예요. 이분 탐색을 곁들이면 O(N log N)까지 줄어드는데, 길이만 필요할 때 자주 써요.

LCS : 최장 공통 부분 수열

두 문자열에서 순서를 지키며 공통으로 뽑을 수 있는 가장 긴 부분 수열이에요. 2차원 표 L[i, j] 를 채워요.

int[,] L = new int[n + 1, m + 1];
for (int i = 1; i <= n; i++)
    for (int j = 1; j <= m; j++)
        L[i, j] = (A[i - 1] == B[j - 1])
            ? L[i - 1, j - 1] + 1            // 글자가 같으면 대각선 + 1
            : Math.Max(L[i - 1, j], L[i, j - 1]); // 다르면 위/왼쪽 중 큰 값

0/1 배낭 (Knapsack)

무게 한도 W 안에서 물건을 골라 가치 합을 최대로 만드는 문제예요. 물건은 통째로 담거나 안 담거나 둘 중 하나예요. 1차원 배열을 뒤에서 앞으로 갱신하는 게 핵심이에요.

int[] dp = new int[W + 1];
for (int i = 0; i < n; i++)
    for (int w = W; w >= weight[i]; w--)  // 뒤에서부터: 한 물건을 두 번 담지 않으려고
        dp[w] = Math.Max(dp[w], dp[w - weight[i]] + value[i]);
// dp[W] 가 최대 가치예요
배낭에서 자주 하는 실수 : 무게 루프를 앞에서부터(작은 w 에서 큰 w) 돌면 같은 물건을 여러 번 담게 돼요(그건 무한 배낭이에요). 0/1 배낭은 반드시 큰 w에서 작은 w로 내려가며 갱신하세요.
  • 이럴 때 써요
  • LIS : 가장 긴 증가/감소 구간, 줄 세우기, 겹치지 않게 고르기
  • LCS : 두 수열의 유사도, 편집 거리의 토대
  • 배낭 : 한정된 용량이나 예산에서 가치 최대화, 부분합으로 만들 수 있는 값 판별

한눈에 정리

  • 언제 : 작은 문제의 답이 큰 문제에 그대로 쓰일 때예요.
  • 복잡도 : LIS O(N^2) 또는 O(N log N), LCS O(NM), 배낭 O(NW).
  • 핵심 : dp 의 의미(무엇을 담는 칸인지)를 한 문장으로 정의하는 게 먼저예요.
  • 실수 주의 : 0/1 배낭의 무게 루프 방향, 표 인덱스의 1 밀림에 주의하세요.
QUIZ 10/1 배낭에서 1차원 dp 배열의 무게 루프를 큰 값에서 작은 값으로 도는 이유는?
정답: 한 물건을 한 번만 담으려고
작은 무게부터 갱신하면 방금 이 물건을 담아 갱신한 칸을 같은 물건이 또 참고해 중복으로 담겨요. 큰 무게에서 작은 무게로 내려가면 각 물건이 한 번만 반영돼 0/1 조건을 지켜요.
QUIZ 2LCS 에서 두 글자 A[i-1] 과 B[j-1] 이 같을 때 L[i, j] 는 무엇이 될까요?
정답: L[i-1, j-1] + 1
글자가 같으면 그 글자를 공통 수열에 새로 더할 수 있어요. 그래서 두 문자열을 한 칸씩 줄인 상태의 답(대각선 위 L[i-1, j-1])에 1을 더해요. 다를 때만 위나 왼쪽 중 큰 값을 가져와요.
그래프

최소 신장 트리 (크루스칼 / 프림)

모든 도시를 잇되 도로 길이의 합을 가장 적게 만드는 문제예요. 모든 노드를 사이클 없이 연결하는 가장 싼 트리를 찾아요.

최소 신장 트리(MST)란

그래프의 모든 노드를 잇는 부분 그래프 중, 간선 가중치 합이 최소인 트리예요. 노드가 N개면 간선은 정확히 N-1개를 골라요. 만드는 방법은 두 가지가 대표적이에요.

크루스칼 : 싼 간선부터 욕심내기

모든 간선을 가중치 오름차순으로 정렬한 뒤, 사이클을 만들지 않는 간선만 차례로 더해요. 사이클 판별은 분리 집합(Union-Find)으로 해요.

// edges 를 가중치 오름차순으로 정렬해 두었다고 할게요
Array.Sort(edges, (x, y) => x.w - y.w);

int total = 0, used = 0;
foreach (var e in edges)
{
    if (Find(e.u) != Find(e.v))    // 두 끝이 다른 그룹일 때만
    {
        Union(e.u, e.v);
        total += e.w;
        if (++used == n - 1) break;  // 간선 N-1개를 골랐으면 끝
    }
}
분리 집합과 한 몸이에요 : "이 간선을 더하면 사이클이 생기나?"는 Find(u) == Find(v) 한 줄로 판별돼요. 14번 주제(Union-Find)를 먼저 익히면 크루스칼은 거저예요.

프림 : 한 점에서 키워 나가기

시작 노드 하나에서 출발해, 지금 트리에 닿는 간선 중 가장 싼 것을 우선순위 큐로 골라 붙여 나가요. 다익스트라와 모양이 거의 같아요.

// pq: (노드, 가중치). 트리에 새로 닿는 가장 싼 간선을 꺼내요
var pq = new PriorityQueue<int, int>();
pq.Enqueue(start, 0);
int total = 0;
while (pq.Count > 0)
{
    pq.TryDequeue(out int u, out int w);
    if (visited[u]) continue;
    visited[u] = true;
    total += w;
    foreach (var (v, c) in adj[u])
        if (!visited[v]) pq.Enqueue(v, c);
}
  • 이럴 때 써요
  • "모든 지점을 최소 비용으로 연결" (도로, 전선, 통신망 설치)
  • 간선이 적으면(희소) 크루스칼, 간선이 빽빽하면 프림이 보통 편해요

한눈에 정리

  • 언제 : 모든 노드를 가장 싸게 연결할 때예요.
  • 복잡도 : 크루스칼 O(E log E)(정렬이 지배), 프림 O(E log V).
  • 핵심 : 크루스칼은 간선 정렬 + 사이클 거르기, 프림은 우선순위 큐로 가장 싼 연결 고르기.
  • 실수 주의 : 간선 N-1개를 고르면 멈추세요. 크루스칼에서 Union-Find 없이 사이클을 못 거르면 틀려요.
QUIZ 1크루스칼에서 정렬된 간선을 더할 때 건너뛰어야 하는 경우는 언제일까요?
정답: 이미 같은 그룹일 때
Find(u) == Find(v) 이면 두 노드는 이미 연결돼 있어요. 여기에 간선을 또 더하면 사이클이 생겨 트리가 깨져요. 그래서 서로 다른 그룹일 때만 잇고 Union 으로 합쳐요.
QUIZ 2노드가 N개일 때 최소 신장 트리가 고르는 간선의 수는 몇 개일까요?
정답: N - 1개
모든 노드를 사이클 없이 연결하는 트리이므로 간선은 항상 N-1개예요. 그래서 크루스칼은 N-1개를 고르는 순간 멈추면 돼요.
그래프

플로이드-워셜 (전체 쌍 최단 경로)

모든 노드에서 모든 노드까지의 최단 거리를 한 번에 구해요. 노드가 적을 때(보통 수백 개 이하) 세 줄짜리 3중 루프로 끝나는 게 매력이에요.

거쳐 가는 노드를 하나씩 허락하기

핵심 아이디어는 "i에서 j로 갈 때 노드 k를 거쳐 가면 더 짧아지나?"를 모든 k에 대해 시험하는 거예요. dist[i, j] 를 i에서 j까지 현재까지의 최단 거리로 두고 갱신해요.

// dist[i,j]: 간선 가중치로 초기화, 없으면 INF, dist[i,i] = 0
for (int k = 0; k < n; k++)          // 거쳐 가는 노드 (가장 바깥!)
    for (int i = 0; i < n; i++)
        for (int j = 0; j < n; j++)
            if (dist[i, k] + dist[k, j] < dist[i, j])
                dist[i, j] = dist[i, k] + dist[k, j];
루프 순서가 생명이에요 : 거쳐 가는 노드 k 가 반드시 가장 바깥 루프여야 해요. i, j 를 바깥에 두면 아직 준비되지 않은 경로를 참고해 틀린 답이 나와요.
오버플로 주의 : 도달 불가를 큰 수 INF로 둘 때 dist[i,k] + dist[k,j] 가 INF + INF 로 넘칠 수 있어요. INF 를 너무 크지 않게(예: 십억대) 잡거나, 두 값이 INF 면 건너뛰세요.

동작 한 줄 요약

k 루프가 한 바퀴 끝날 때마다 "0번부터 k번 노드까지만 중간 경유지로 허락한 최단 거리"가 완성돼요. k 가 n-1까지 끝나면 모든 경유가 허락된 진짜 최단 거리예요.

  • 이럴 때 써요
  • 노드 수가 작고(수백 이하) 모든 쌍의 최단 거리가 필요할 때
  • 임의의 두 지점 사이 거리를 여러 번 물어볼 때
  • 경유 가능 여부(도달성)만 1과 0으로 채우는 변형에도 그대로 써요

한눈에 정리

  • 언제 : 전체 쌍 최단 거리, 노드 수가 작을 때예요.
  • 복잡도 : O(N^3). N이 500 근처를 넘으면 부담돼요.
  • 핵심 : 경유 노드 k 를 가장 바깥 루프에 두고 dist[i,k] + dist[k,j] 로 갱신.
  • 실수 주의 : 루프 순서(k 바깥)와 INF 덧셈 오버플로를 조심하세요.
QUIZ 1플로이드-워셜에서 가장 바깥에 와야 하는 루프는 무엇일까요?
정답: 거쳐 가는 노드 k
k 를 바깥에 두어야 "0번부터 k번까지를 경유지로 허락한 최단 거리"가 단계적으로 완성돼요. i 나 j 를 바깥에 두면 아직 갱신되지 않은 거리를 참고해 틀린 답이 나와요.
QUIZ 2노드가 400개인 그래프에서 플로이드-워셜의 대략적인 연산량은?
정답: 약 6400만 번
3중 루프라 O(N^3)이에요. 400^3 은 약 6400만으로 충분히 빠르지만, N 이 1000을 넘으면 10억대가 되어 시간이 빠듯해요. 그땐 다익스트라를 여러 번 돌리는 편이 나아요.
그래프

벨만-포드 (음수 간선 최단 경로)

다익스트라가 못 푸는, 음수 간선이 있는 그래프의 최단 거리를 구해요. 게다가 음수 사이클(돌수록 비용이 줄어드는 고리)까지 잡아내요.

모든 간선을 V-1번 완화하기

완화(relax)란 "u를 거쳐 v로 가면 더 짧아지나?"를 확인해 거리를 줄이는 거예요. 모든 간선을 한 바퀴 완화하길 노드 수보다 하나 적게(V-1번) 반복하면 최단 거리가 확정돼요.

long[] dist = new long[n];
Array.Fill(dist, INF);
dist[start] = 0;

for (int i = 0; i < n - 1; i++)        // V-1번 반복
    foreach (var (u, v, w) in edges)
        if (dist[u] != INF && dist[u] + w < dist[v])
            dist[v] = dist[u] + w;
왜 V-1번일까요 : 최단 경로는 노드를 많아야 V개 지나니 간선은 최대 V-1개예요. 한 번 완화할 때마다 경로가 간선 하나씩 길어질 수 있으니, V-1번이면 가장 긴 최단 경로까지 완성돼요.

음수 사이클 판별

V-1번을 다 돌린 뒤 한 번 더 완화해서 또 줄어드는 간선이 있다면, 돌수록 비용이 작아지는 음수 사이클이 있다는 뜻이에요.

bool hasNegativeCycle = false;
foreach (var (u, v, w) in edges)
    if (dist[u] != INF && dist[u] + w < dist[v])
        hasNegativeCycle = true;   // 한 번 더 줄어들면 음수 사이클
다익스트라와의 갈림길 : 음수 간선이 없으면 빠른 다익스트라(O(E log V))를 쓰세요. 벨만-포드는 O(VE)로 느리지만, 음수 간선이나 음수 사이클 판별이 필요할 때만의 무기예요.
  • 이럴 때 써요
  • 간선 가중치에 음수가 섞여 있을 때
  • "무한히 이득을 보는 고리가 있는가?" 같은 음수 사이클 판별
  • 환전 차익이나 시간 되감기처럼 비용이 줄어드는 모델

한눈에 정리

  • 언제 : 음수 간선이 있거나 음수 사이클을 찾을 때예요.
  • 복잡도 : O(V x E). 다익스트라보다 느려요.
  • 핵심 : 모든 간선을 V-1번 완화, 한 번 더 줄면 음수 사이클.
  • 실수 주의 : 아직 못 간 노드(dist == INF)에서 완화하지 마세요. 거리 합은 long 으로 두세요.
QUIZ 1벨만-포드에서 모든 간선을 V-1번 완화하는 이유로 알맞은 것은?
정답: 최단 경로의 간선이 최대 V-1개라서
사이클이 없는 최단 경로는 노드를 많아야 V개 지나니 간선은 최대 V-1개예요. 한 번 완화에 경로가 간선 하나씩 길어질 수 있으므로 V-1번이면 가장 긴 최단 경로까지 모두 확정돼요.
QUIZ 2V-1번 완화를 끝낸 뒤 한 번 더 돌렸더니 또 줄어드는 간선이 있었어요. 무슨 뜻일까요?
정답: 음수 사이클이 있어요
V-1번이면 최단 거리가 확정돼야 정상이에요. 그런데도 더 줄어든다면 돌 때마다 비용이 작아지는 음수 사이클이 있다는 신호예요. 이 경우 최단 거리 자체가 정의되지 않아요.
문자열

문자열 매칭 (KMP / 라빈-카프)

긴 글 속에서 특정 패턴이 어디에 나오는지 빠르게 찾는 방법이에요. 한 글자씩 다 비교하면 느린데, 이미 비교한 정보를 재활용해 O(N + M)으로 줄여요.

단순 비교는 왜 느릴까

본문 길이 N, 패턴 길이 M일 때 모든 위치에서 처음부터 다시 비교하면 O(N x M)이에요. 본문과 패턴이 길면 시간이 폭발해요. 핵심은 "틀렸을 때 처음으로 돌아가지 않는 것"이에요.

KMP : 실패 함수로 건너뛰기

패턴 안에서 "접두사이면서 접미사인 가장 긴 길이"를 미리 표(실패 함수)로 만들어 둬요. 비교하다 틀리면 처음이 아니라 이 표가 알려주는 위치로 점프해, 본문 인덱스를 절대 되돌리지 않아요.

int[] BuildPi(string p)   // 실패 함수: 접두사이자 접미사인 최대 길이
{
    int[] pi = new int[p.Length];
    int j = 0;
    for (int i = 1; i < p.Length; i++)
    {
        while (j > 0 && p[i] != p[j]) j = pi[j - 1];
        if (p[i] == p[j]) pi[i] = ++j;
    }
    return pi;
}

// 본문 s 에서 패턴 p 찾기 (O(N + M))
void Kmp(string s, string p)
{
    int[] pi = BuildPi(p);
    int j = 0;
    for (int i = 0; i < s.Length; i++)
    {
        while (j > 0 && s[i] != p[j]) j = pi[j - 1];
        if (s[i] == p[j] && ++j == p.Length)
        {
            // i - p.Length + 1 위치에서 발견
            j = pi[j - 1];
        }
    }
}

라빈-카프 : 해시로 한 번에 비교

패턴의 해시 값을 미리 구하고, 본문에서 같은 길이 구간의 해시를 한 칸씩 밀며(롤링 해시) 빠르게 갱신해 비교해요. 해시가 같을 때만 실제 문자열을 확인하면 평균 O(N + M)이에요. 여러 패턴을 동시에 찾을 때 특히 편해요.

  • 이럴 때 써요
  • 긴 텍스트에서 부분 문자열의 등장 위치나 횟수를 셀 때
  • 문자열이 다른 문자열의 부분인지 빠르게 확인할 때
  • 여러 패턴을 한꺼번에 검색할 때 (라빈-카프)

한눈에 정리

  • 언제 : 긴 본문에서 패턴을 찾을 때예요.
  • 복잡도 : 단순 비교 O(NM), KMP/라빈-카프 O(N + M).
  • 핵심 : KMP는 실패 함수로 본문 인덱스를 되돌리지 않아요.
  • 실수 주의 : 실패 함수는 패턴 자신을 패턴으로 비교해 만들어요. 라빈-카프는 해시 충돌 시 실제 비교로 확인하세요.
QUIZ 1KMP의 실패 함수(부분 일치 테이블)가 담는 값은 무엇일까요?
정답: 접두사이자 접미사인 최대 길이
이 값을 알면 비교가 틀렸을 때 처음으로 돌아가지 않고, 이미 일치한 접두사만큼 건너뛴 위치에서 다시 비교할 수 있어요. 덕분에 본문 인덱스를 되돌리지 않아 O(N + M)이 돼요.
QUIZ 2본문 길이 N, 패턴 길이 M일 때 단순 비교와 KMP의 시간 복잡도는?
정답: O(NM) vs O(N + M)
단순 비교는 매 위치에서 패턴을 처음부터 다시 맞춰 최악 O(NM)이에요. KMP는 실패 함수로 겹치는 비교를 없애 O(N + M)으로 끝나요.
자료구조

트라이 (Trie)

문자열들을 글자 단위로 잇는 트리예요. "이 접두사로 시작하는 단어가 있나?"를 단어 길이에 비례하는 시간에 답해, 자동완성과 사전에 딱이에요.

구조

각 노드는 다음 글자로 가는 자식들을 가지고, 단어가 끝나는 노드에 표시(end)를 남겨요. 루트에서 글자를 따라 내려가면 한 단어가 돼요.

class Trie
{
    Dictionary<char, Trie> children = new();
    bool end;

    public void Insert(string w)
    {
        var node = this;
        foreach (char c in w)
        {
            if (!node.children.ContainsKey(c)) node.children[c] = new Trie();
            node = node.children[c];
        }
        node.end = true;     // 여기서 한 단어가 끝나요
    }

    public bool Search(string w)
    {
        var node = this;
        foreach (char c in w)
            if (!node.children.TryGetValue(c, out node)) return false;
        return node.end;
    }
}
접두사 검색은 거의 공짜예요 : Search 에서 마지막 end 검사만 빼면 "이 접두사로 시작하는 단어가 있나?"가 돼요. 자동완성이 바로 이거예요.
  • 이럴 때 써요
  • 자동완성, 사전 검색, 접두사로 묶어 세기
  • 여러 단어 중 공통 접두사를 다룰 때
  • 비트 트라이로 확장하면 XOR 최대값 같은 문제도 풀려요

한눈에 정리

  • 언제 : 문자열 집합에서 접두사 질의가 잦을 때예요.
  • 복잡도 : 삽입과 검색 모두 단어 길이 L에 비례하는 O(L).
  • 핵심 : 글자를 따라 내려가고, 단어 끝 노드에 표시를 남겨요.
  • 실수 주의 : 단어 끝 표시(end)가 없으면 접두사와 완성된 단어를 구분 못 해요. 메모리를 많이 쓰니 글자 종류가 적을 때 유리해요.
QUIZ 1트라이에서 단어 하나를 검색하는 시간 복잡도는?
정답: O(L)
저장된 단어가 아무리 많아도, 검색은 찾는 단어의 글자를 하나씩 따라 내려가기만 하면 돼요. 그래서 단어 길이 L에만 비례해요.
QUIZ 2노드에 "단어 끝(end)" 표시를 두는 이유는 무엇일까요?
정답: 접두사와 완성 단어 구분
"car"를 넣었을 때 "ca"까지 가는 경로도 생겨요. end 표시가 없으면 "ca"도 등록된 단어로 착각해요. 끝 노드에만 표시를 남겨야 정확히 구분돼요.
자료구조

세그먼트 트리와 펜윅 트리

구간의 합이나 최솟값을 빠르게 구하면서, 값도 자주 바꿔야 할 때 쓰는 자료구조예요. 질의와 갱신을 모두 O(log N)에 처리해요.

누적 합만으론 부족할 때

구간 합은 누적 합으로 O(1)에 구하지만, 중간 값이 바뀌면 누적 합을 다시 만드는 데 O(N)이 들어요. 갱신이 잦으면 이게 발목을 잡아요. 그래서 질의도 갱신도 O(log N)인 트리형 구조가 필요해요.

펜윅 트리(BIT) : 짧고 빠른 구간 합

마지막 켜진 비트(i & (-i))를 이용해 부분 합을 층층이 저장해요. 코드가 아주 짧아 구간 합 문제에 자주 써요.

long[] tree;   // 1-indexed, 크기 n+1

void Update(int i, long delta)   // i번째 원소에 delta 더하기
{
    for (; i <= n; i += i & (-i)) tree[i] += delta;
}

long Query(int i)              // 1..i 구간 합
{
    long s = 0;
    for (; i > 0; i -= i & (-i)) s += tree[i];
    return s;
}
// 구간 [l, r] 합 = Query(r) - Query(l - 1)

세그먼트 트리 : 더 일반적인 구간 질의

구간을 반씩 쪼갠 트리를 만들어, 각 노드가 자기 구간의 합(또는 최솟값, 최댓값)을 들고 있어요. 펜윅보다 코드가 길지만 최솟값, 최댓값, 구간 갱신처럼 더 다양한 질의를 다룰 수 있어요.

  • 이럴 때 써요
  • 값이 자주 바뀌는데 구간 합이나 최솟값을 반복해 물어볼 때
  • "i번째를 v로 바꾼 뒤 a부터 b까지 합" 같은 질의가 섞여 나올 때
  • 단순 합만 필요하면 펜윅, 최솟값 등 일반 질의면 세그먼트 트리

한눈에 정리

  • 언제 : 갱신과 구간 질의가 둘 다 잦을 때예요.
  • 복잡도 : 질의와 갱신 모두 O(log N).
  • 핵심 : 펜윅은 짧은 구간 합 전용, 세그먼트 트리는 더 일반적.
  • 실수 주의 : 펜윅은 1-indexed로 쓰세요(0이면 무한 루프). 구간 합은 Query(r) - Query(l-1)로 빼서 구해요.
QUIZ 1값이 자주 바뀌는데도 누적 합 대신 펜윅 트리를 쓰는 이유는?
정답: 누적 합의 갱신이 O(N)이라서
누적 합은 질의는 O(1)이지만 중간 값이 바뀌면 뒤쪽 누적 값을 전부 고쳐야 해 O(N)이에요. 펜윅 트리는 갱신과 질의를 모두 O(log N)에 처리해 갱신이 잦은 상황에 유리해요.
QUIZ 2펜윅 트리로 구간 [l, r]의 합을 구하는 식으로 알맞은 것은?
정답: Query(r) - Query(l - 1)
Query(i)는 1부터 i까지의 합이에요. [l, r] 합은 1..r 합에서 1..(l-1) 합을 빼면 정확히 남아요. l을 빼면 l번째 원소까지 사라지니 l-1로 빼야 해요.
알고리즘 설계

분할 정복

큰 문제를 작은 문제 여러 개로 쪼개 각각 풀고, 그 답을 합쳐 전체를 푸는 전략이에요. 병합 정렬, 빠른 거듭제곱이 대표 예예요.

세 단계 : 나누고, 풀고, 합치기

문제를 절반으로 나눠 재귀로 풀고(정복), 부분 답을 합쳐(결합) 전체 답을 만들어요. 보통 깊이가 log N이라 한 단계의 합치는 비용에 log N을 곱한 만큼 걸려요.

병합 정렬이 전형적이에요. 배열을 반으로 나눠 각각 정렬한 뒤, 두 정렬된 조각을 앞에서부터 비교하며 합쳐 O(N log N)에 정렬해요.

빠른 거듭제곱

a를 n번 곱하면 O(n)이지만, "지수를 절반으로 줄이며 제곱"하면 O(log n)에 끝나요. 큰 수 나머지 계산에 꼭 나와요.

// a^n mod m 을 O(log n) 에
long Power(long a, long n, long mod)
{
    long result = 1; a %= mod;
    while (n > 0)
    {
        if ((n & 1) == 1) result = result * a % mod;  // 지수의 끝 비트가 1이면 곱하기
        a = a * a % mod;   // 밑을 제곱
        n >>= 1;          // 지수를 절반으로
    }
    return result;
}
  • 이럴 때 써요
  • 정렬(병합 정렬), 큰 지수의 거듭제곱
  • "가까운 두 점", "역수열(inversion) 세기"처럼 반으로 나눠 합칠 때
  • 거듭제곱 아이디어를 행렬에 적용하면 점화식도 O(log n)에 계산해요

한눈에 정리

  • 언제 : 문제를 절반으로 쪼개 풀고 합칠 수 있을 때예요.
  • 복잡도 : 흔히 O(N log N)(병합 정렬), O(log n)(빠른 거듭제곱).
  • 핵심 : 나누기 -> 정복(재귀) -> 합치기. 합치는 비용이 전체를 좌우해요.
  • 실수 주의 : 거듭제곱에서 곱셈마다 mod 를 취하고 long 을 쓰세요. 재귀 종료 조건(크기 1)을 빠뜨리지 마세요.
QUIZ 1빠른 거듭제곱으로 a^n 을 계산하는 시간 복잡도는?
정답: O(log n)
지수를 매번 절반으로 줄이며 밑을 제곱하니, 반복 횟수가 n의 비트 수, 즉 log n 번이에요. n이 십억이어도 서른 번 정도면 끝나요.
QUIZ 2병합 정렬에서 "합치기(merge)" 단계가 하는 일은 무엇일까요?
정답: 정렬된 두 조각을 합친다
각 절반은 재귀로 이미 정렬돼 있어요. 두 조각의 앞쪽을 비교해 작은 것부터 차례로 꺼내면 O(N)에 합쳐져요. 이 합치기가 분할 정복의 결합 단계예요.
동적 계획법

DP 심화 (트리 / 구간 / 비트마스크)

기본 DP를 넘어, 상태를 트리 위에 두거나 구간으로 잡거나 집합을 비트로 압축하는 세 가지 심화 패턴이에요. 골드 이상에서 자주 나와요.

트리 DP : 자식 결과를 모아 부모를 계산

후위 순회로 자식의 답을 먼저 구한 뒤 부모를 채워요. 대표 예는 트리에서 서로 인접하지 않게 고른 최대 노드 수(독립 집합)예요.

int[,] dp;   // dp[u,0]=u 안 뽑음, dp[u,1]=u 뽑음
void Dfs(int u, int parent)
{
    dp[u, 0] = 0; dp[u, 1] = 1;
    foreach (int v in adj[u])
    {
        if (v == parent) continue;
        Dfs(v, u);                                  // 자식부터 (후위)
        dp[u, 0] += Math.Max(dp[v, 0], dp[v, 1]);
        dp[u, 1] += dp[v, 0];                      // 나를 뽑으면 자식은 못 뽑음
    }
}

구간 DP : 작은 구간부터 넓혀 가기

dp[i, j] 를 "구간 i부터 j까지의 답"으로 두고, 길이가 짧은 구간부터 채워요. 행렬 곱셈 순서, 돌 합치기 같은 문제가 여기 속해요. 구간을 둘로 가르는 지점 k를 모두 시험해요.

비트마스크 DP : 집합을 비트로

방문한 원소의 집합을 정수의 비트로 표현해요. 대표 예가 외판원 문제(TSP)로, "어떤 도시들을 방문했고 지금 어디에 있나"를 (mask, last)로 저장해요.

// dp[mask, last] = mask 도시들을 방문하고 last 에서 끝낸 최소 비용
for (int mask = 0; mask < (1 << n); mask++)
    for (int last = 0; last < n; last++)
    {
        if ((mask & (1 << last)) == 0) continue;   // last 가 mask 에 없으면 건너뛰기
        // 다음 도시 next 로 확장하며 dp[mask | (1 << next), next] 갱신
    }
비트마스크 DP의 한계 : 상태가 2^n 개라 n이 커지면 폭발해요. 보통 n이 20 안팎(원소 스무 개 이하)일 때만 써요.
  • 이럴 때 써요
  • 트리 DP : 트리 구조에서 부분 트리마다 답을 모을 때
  • 구간 DP : 구간을 합치는 순서나 비용이 문제일 때
  • 비트마스크 DP : 원소가 적고(스무 개 이하) 방문 집합을 기억해야 할 때

한눈에 정리

  • 언제 : 상태를 트리, 구간, 집합으로 잡아야 할 때예요.
  • 복잡도 : 트리 DP O(N), 구간 DP O(N^3) 안팎, 비트마스크 DP O(2^n x n).
  • 핵심 : dp 의 상태(무엇을 기억하는 칸인지)를 정확히 정의하는 게 먼저예요.
  • 실수 주의 : 트리 DP는 부모로 되돌아가지 않게, 비트마스크 DP는 n 범위(보통 20 이하)를 확인하세요.
QUIZ 1트리 DP에서 보통 후위 순회(자식 먼저)로 계산하는 이유는?
정답: 부모가 자식 답에 의존해서
트리 DP는 자식의 결과를 모아 부모를 정해요. 그래서 자식을 먼저 다 계산하는 후위 순회로 내려갔다 올라오며 채워야 부모를 올바르게 구할 수 있어요.
QUIZ 2비트마스크 DP를 보통 원소 20개 안팎까지만 쓰는 이유는?
정답: 상태가 2^n 개라서
방문 집합을 비트로 표현하면 가능한 집합이 2^n 가지예요. n이 20이면 약 100만, 25면 3천만이 넘어 빠르게 한계에 닿아요. 그래서 원소가 적을 때만 쓰는 기법이에요.
수학

조합론

경우의 수를 세는 도구예요. 순열과 조합 공식, 파스칼 삼각형, 그리고 큰 수를 다루는 모듈러 조합까지가 자주 나와요.

순열과 조합

서로 다른 n개에서 r개를 순서 있게 뽑으면 순열 P(n, r) = n! / (n-r)!, 순서 없이 뽑으면 조합 C(n, r) = n! / (r! (n-r)!)이에요. "줄 세우기"는 순열, "고르기"는 조합이라고 기억하면 쉬워요.

파스칼 삼각형으로 조합 구하기

C(n, r) = C(n-1, r-1) + C(n-1, r)이라는 성질로, 큰 수 나눗셈 없이 표를 채워 조합을 구해요. 나머지(MOD) 연산과도 잘 어울려요.

long[,] C = new long[n + 1, n + 1];
for (int i = 0; i <= n; i++)
{
    C[i, 0] = 1;
    for (int j = 1; j <= i; j++)
        C[i, j] = (C[i - 1, j - 1] + C[i - 1, j]) % MOD;  // 위 두 칸의 합
}

모듈러 조합 (나눗셈은 역원으로)

답을 소수 MOD로 나눈 나머지로 요구할 때가 많아요. 나머지 세계에서는 나눗셈을 바로 못 하니, 페르마 소정리로 분모의 역원을 구해 곱해요. 역원은 빠른 거듭제곱으로 분모^(MOD-2) % MOD 예요.

나머지에서 나눗셈 주의 : (a / b) % MOD 는 틀려요. a % MOD x (b의 역원) % MOD 로 계산해야 해요. 역원은 MOD 가 소수일 때 b^(MOD-2)로 구해요.
  • 이럴 때 써요
  • "경우의 수를 1000000007로 나눈 나머지" 같은 문제
  • 줄 세우기(순열), 팀 고르기(조합), 격자 경로 수
  • 이항 계수가 반복해서 필요할 때 파스칼 표로 미리 계산

한눈에 정리

  • 언제 : 경우의 수를 셀 때, 특히 큰 수의 나머지를 요구할 때예요.
  • 복잡도 : 파스칼 표 O(N^2), 역원 한 번 O(log MOD).
  • 핵심 : C(n, r) = C(n-1, r-1) + C(n-1, r), 나머지 나눗셈은 역원으로.
  • 실수 주의 : 나머지 세계에서 나눗셈을 그대로 쓰지 마세요. 곱셈 중간마다 MOD 를 취하고 long 을 쓰세요.
QUIZ 1파스칼 삼각형에서 C(n, r)을 만드는 점화식으로 알맞은 것은?
정답: C(n-1, r-1) + C(n-1, r)
n번째 원소를 뽑는 경우(C(n-1, r-1))와 안 뽑는 경우(C(n-1, r))로 갈라 더하면 C(n, r)이 돼요. 이 성질 덕에 나눗셈 없이 표로 조합을 구할 수 있어요.
QUIZ 2소수 MOD로 나눈 나머지 세계에서 b로 나누려면 어떻게 해야 할까요?
정답: 역원을 곱한다
나머지 세계에서는 나눗셈이 곧 "역원 곱하기"예요. MOD 가 소수면 페르마 소정리로 b의 역원이 b^(MOD-2) % MOD 이고, 이걸 빠른 거듭제곱으로 구해 곱하면 정확한 답이 나와요.
수학

모듈러 연산

답이 너무 커서 "어떤 수로 나눈 나머지를 출력하라"는 문제가 많아요. 나머지 세계의 계산 규칙을 알면 큰 수도 넘침 없이 다룰 수 있어요.

기본 규칙

더하기와 곱하기는 중간에 나머지를 취해도 결과가 같아요. 그래서 큰 수가 되기 전에 그때그때 나머지를 취해 넘침을 막아요. 빼기는 음수가 나올 수 있으니 m을 더해 보정해요.

long add = ((a % m) + (b % m)) % m;
long mul = (a % m) * (b % m) % m;
long sub = ((a - b) % m + m) % m;   // 음수 나머지를 0..m-1 로 보정
곱셈 넘침 주의 : a * b 가 int 범위를 쉽게 넘어요. 나머지 곱은 long 으로 두고 곱한 직후 % m 을 취하세요.

큰 지수는 빠른 거듭제곱으로

a를 n번 곱한 나머지가 필요하면 분할 정복의 빠른 거듭제곱으로 O(log n)에 구해요(29번 주제). 모듈러 역원도 여기서 나와요.

나눗셈은 역원으로

나머지 세계에서는 나눗셈을 바로 못 해요. m이 소수면 페르마 소정리로 b의 역원이 b^(m-2) mod m 이고, 나누는 대신 이 역원을 곱해요.

// a / b mod m = a * Inverse(b) mod m   (m 이 소수일 때)
long Inverse(long b, long m) => Power(b, m - 2, m);   // 페르마 소정리
  • 이럴 때 써요
  • "답을 1000000007로 나눈 나머지" 류의 큰 수 문제
  • 조합론, 경우의 수, 큰 피보나치 등 결과가 폭발하는 계산
  • 해시(라빈-카프)에서 롤링 해시 값을 다룰 때

한눈에 정리

  • 언제 : 결과가 매우 커서 나머지를 요구할 때예요.
  • 복잡도 : 곱셈마다 O(1), 역원 한 번 O(log m).
  • 핵심 : 더하기/곱하기는 중간 나머지 OK, 나누기는 역원 곱하기.
  • 실수 주의 : long 사용, 빼기 음수 보정, 나눗셈을 그냥 쓰지 않기.
QUIZ 1나머지 세계에서 그대로 적용하면 안 되는 연산은 무엇일까요?
정답: 나눗셈
덧셈과 곱셈은 중간에 나머지를 취해도 결과가 같아요. 하지만 나눗셈은 그대로 못 하고, m이 소수일 때 b의 역원 b^(m-2)를 곱해야 정확한 답이 나와요.
QUIZ 2뺄셈 (a - b) % m 의 결과가 음수가 나올 때 올바른 보정은?
정답: ((a - b) % m + m) % m
프로그래밍 언어의 나머지는 음수가 나올 수 있어요. m을 한 번 더한 뒤 다시 % m 을 취하면 항상 0부터 m-1 사이의 올바른 나머지로 맞춰져요.
기법

좌표 압축

값의 범위는 수십억으로 넓은데 실제 등장하는 값은 몇 개뿐일 때, 값을 "순위"로 바꿔 작은 인덱스로 다루는 기법이에요.

왜 필요할까

값이 최대 10억인데 개수는 10만 개라면, 값 자체를 배열 인덱스로 쓰는 건 불가능해요(크기 10억 배열). 하지만 등장하는 값은 10만 개뿐이니, 이들을 정렬해 0, 1, 2... 순위로 바꾸면 작은 배열로 다룰 수 있어요. 대소 관계는 그대로 보존돼요.

방법 : 정렬 + 중복 제거 + 순위 찾기

// 1) 등장한 값들을 정렬하고 중복을 제거
var sorted = arr.Distinct().OrderBy(x => x).ToArray();

// 2) 값 v 를 그 순위(인덱스)로 바꾸기
int Rank(int v) => Array.BinarySearch(sorted, v);

// 이제 arr[i] 대신 Rank(arr[i]) 를 쓰면 0..k-1 의 작은 인덱스
순서만 바뀌지 않아요 : 압축해도 "누가 더 큰가"는 그대로예요. 그래서 정렬, 구간 합, 펜윅 트리처럼 대소와 인덱스만 중요한 문제에 안전하게 써요.
  • 이럴 때 써요
  • 값의 범위가 너무 넓어 배열이나 펜윅 트리를 못 잡을 때
  • 등장하는 값의 개수는 적을 때
  • 구간 질의, 스위핑, 펜윅 트리와 함께 자주 쓰여요

한눈에 정리

  • 언제 : 값 범위는 넓고 개수는 적을 때예요.
  • 복잡도 : 정렬 O(N log N), 순위 찾기 O(log N).
  • 핵심 : 정렬 + 중복 제거 후 이분 탐색으로 순위를 매겨요.
  • 실수 주의 : 중복 제거를 빠뜨리면 순위가 어긋나요. 원래 값이 필요할 땐 sorted 배열로 되돌리세요.
QUIZ 1좌표 압축을 쓰기에 가장 알맞은 상황은?
정답: 범위는 넓고 개수는 적을 때
값이 0부터 100처럼 촘촘하면 압축할 이득이 없어요. 범위가 넓어 배열을 못 잡는데 실제 값은 적을 때, 순위로 바꾸면 작은 인덱스로 다룰 수 있어 효과가 커요.
QUIZ 2좌표 압축에서 정렬한 값 배열에 중복 제거가 필요한 이유는?
정답: 같은 값은 같은 순위여야 해서
중복을 남기면 동일한 값이 여러 인덱스로 흩어져 순위가 어긋나요. 중복을 제거해야 각 값이 유일한 순위 하나에 대응돼 압축이 올바르게 동작해요.
기법

스위핑 (라인 스윕)

시작과 끝 같은 이벤트를 한 줄로 정렬해 한 방향으로 쓸어가며 처리하는 기법이에요. 구간 겹침, 동시 진행 수, 일정 충돌 문제에 강해요.

이벤트로 바꿔 정렬하기

각 구간 [s, e)를 "s에서 +1, e에서 -1"인 두 이벤트로 쪼개요. 이벤트를 위치 순으로 정렬한 뒤 한 줄로 훑으며 누적하면, 현재 겹친 개수를 실시간으로 알 수 있어요.

// 회의실 최대 동시 사용 수 구하기
var events = new List<(int x, int delta)>();
foreach (var (s, e) in intervals)
{
    events.Add((s, +1));   // 시작: 하나 늘어남
    events.Add((e, -1));   // 끝: 하나 줄어듦
}
events.Sort((a, b) => a.x != b.x ? a.x - b.x : a.delta - b.delta);  // 같은 위치면 끝(-1) 먼저

int cur = 0, max = 0;
foreach (var ev in events)
{
    cur += ev.delta;
    max = Math.Max(max, cur);
}
// max = 동시에 겹친 최대 개수
정렬 순서가 정답을 가른다 : 같은 좌표에서 끝(-1)을 시작(+1)보다 먼저 처리하면 "맞닿은 구간"은 겹치지 않는 것으로 봐요. 문제가 맞닿음을 겹침으로 보는지 아닌지에 맞춰 순서를 정하세요.
  • 이럴 때 써요
  • 회의실 배정, 동시 접속자 수, 구간 겹침 개수
  • 구간들의 합집합 길이, 직사각형 넓이의 합
  • 좌표 압축, 펜윅 트리와 묶어 더 큰 문제로 확장

한눈에 정리

  • 언제 : 구간의 겹침이나 동시 진행 수를 다룰 때예요.
  • 복잡도 : 이벤트 정렬이 지배해 O(N log N).
  • 핵심 : 시작 +1, 끝 -1 이벤트로 만들어 정렬 후 누적해요.
  • 실수 주의 : 같은 좌표에서 +1과 -1의 처리 순서를 문제 조건에 맞추세요.
QUIZ 1스위핑으로 최대 동시 겹침 수를 구할 때, 각 구간을 어떻게 이벤트로 바꿀까요?
정답: 시작 +1, 끝 -1
시작에서 겹침이 하나 늘고 끝에서 하나 줄어요. 이벤트를 위치 순으로 훑으며 누적하면 그 순간 겹친 개수가 나오고, 그 최댓값이 최대 동시 겹침 수예요.
QUIZ 2스위핑의 전체 시간 복잡도를 좌우하는 단계는?
정답: 정렬 O(N log N)
이벤트를 한 줄로 훑는 누적은 O(N)이에요. 그 전에 이벤트를 위치 순으로 정렬하는 비용 O(N log N)이 전체를 좌우해요.

최소 공통 조상 (LCA)

트리에서 두 노드의 가장 가까운 공통 조상을 찾는 문제예요. 두 노드 사이 거리나 경로를 구할 때 토대가 돼요.

기본 방법 : 깊이 맞추고 같이 올라가기

먼저 두 노드의 깊이를 같게 맞춘 뒤, 둘이 만날 때까지 함께 한 칸씩 부모로 올라가요. 깊이와 부모 배열은 DFS나 BFS로 미리 구해 둬요.

// depth[], parent[] 를 루트에서 미리 계산해 둠
int Lca(int u, int v)
{
    while (depth[u] > depth[v]) u = parent[u];   // 깊은 쪽을 끌어올려
    while (depth[v] > depth[u]) v = parent[v];   // 깊이를 맞춤
    while (u != v) { u = parent[u]; v = parent[v]; } // 같이 올라가 만나는 지점
    return u;
}
질의가 많으면 희소 배열 : 위 방법은 한 번에 O(N)이라 질의가 많으면 느려요. 2^k 번째 조상을 미리 표로 만드는 희소 배열(sparse table)을 쓰면 질의당 O(log N)으로 빨라져요.
  • 이럴 때 써요
  • 트리에서 두 노드 사이 거리 (depth[u] + depth[v] - 2 x depth[LCA])
  • 두 노드의 경로 위에 있는지 판별
  • 트리 위 구간 질의, 가족 관계 문제

한눈에 정리

  • 언제 : 트리에서 두 노드의 공통 조상이나 거리가 필요할 때예요.
  • 복잡도 : 기본 O(N) per 질의, 희소 배열이면 전처리 O(N log N) + 질의 O(log N).
  • 핵심 : 깊이를 맞춘 뒤 함께 올라가 만나는 노드예요.
  • 실수 주의 : depth와 parent를 정확히 전처리하세요. 질의가 많으면 반드시 희소 배열로 바꾸세요.
QUIZ 1LCA를 단순하게 구할 때 가장 먼저 하는 일은?
정답: 깊이를 맞춘다
깊이가 다르면 함께 올라가도 같은 높이에서 비교가 안 돼요. 더 깊은 쪽을 먼저 끌어올려 깊이를 같게 만든 뒤, 둘이 만날 때까지 함께 올라가면 그 지점이 LCA예요.
QUIZ 2LCA 질의가 매우 많을 때 질의당 O(log N)으로 줄이는 방법은?
정답: 희소 배열(sparse table)
각 노드의 1, 2, 4, 8... 번째 조상을 미리 표로 만들어 두면, 깊이 차이를 이진수로 한 번에 점프해 질의당 O(log N)에 LCA를 구할 수 있어요.
수학

기하 기초 (CCW)

세 점이 어느 방향으로 도는지(반시계, 시계, 일직선)를 외적의 부호로 판별해요. 선분 교차, 볼록 껍질 같은 기하 문제의 출발점이에요.

CCW : 세 점의 방향

두 벡터의 외적 부호로 방향을 알 수 있어요. 양수면 반시계, 음수면 시계, 0이면 한 직선 위에 있어요.

// 세 점 a, b, c 의 방향: 1=반시계, -1=시계, 0=일직선
int Ccw(long ax, long ay, long bx, long by, long cx, long cy)
{
    long cross = (bx - ax) * (cy - ay) - (by - ay) * (cx - ax);
    return cross > 0 ? 1 : (cross < 0 ? -1 : 0);
}
오버플로 주의 : 좌표가 크면 외적의 곱이 int 범위를 넘어요. 좌표와 계산을 long 으로 두세요.

선분 교차 판정

두 선분이 교차하는지는 CCW로 판별해요. 한 선분의 양 끝에서 본 다른 선분 두 끝의 방향이 서로 반대이고, 반대로도 그렇다면 교차해요. 방향이 0인(일직선) 경우만 따로 끝점 포함 여부를 확인하면 돼요.

  • 이럴 때 써요
  • 선분 교차 판정, 점이 다각형 안에 있는지
  • 볼록 껍질(convex hull)을 만들 때 방향 판별
  • 다각형 넓이(신발끈 공식)도 외적에서 나와요

한눈에 정리

  • 언제 : 점과 선분의 방향, 교차를 다루는 기하 문제예요.
  • 복잡도 : CCW 한 번 O(1).
  • 핵심 : 외적 부호로 반시계(+), 시계(-), 일직선(0)을 가려요.
  • 실수 주의 : 좌표 곱이 커지니 long 을 쓰고, 일직선(0) 경우를 따로 처리하세요.
QUIZ 1CCW에서 외적 결과가 0이면 세 점은 어떤 상태일까요?
정답: 일직선
외적이 0이라는 건 두 벡터가 같은 방향(또는 반대 방향)이라 넓이가 0이라는 뜻이에요. 즉 세 점이 한 직선 위에 놓여 있어요. 양수면 반시계, 음수면 시계예요.
QUIZ 2CCW 계산에서 좌표를 long으로 두라고 권하는 이유는?
정답: 곱셈이 넘칠 수 있어서
외적은 좌표 차이끼리의 곱이라, 좌표가 크면 결과가 int 범위를 쉽게 넘어요. 넘치면 부호가 뒤집혀 방향을 틀리게 판단하니 long으로 계산해야 안전해요.
실전

이렇게 묶어서 출제돼요

실제 코딩 테스트 문제는 개념 하나로 끝나지 않고 2~3개를 엮어서 나와요. 자주 묶이는 조합과 "이런 신호가 보이면 이렇게 접근한다"를 알아 두면, 처음 보는 문제도 어떤 개념을 꺼낼지 빠르게 판단할 수 있어요.

읽는 법 : 각 카드의 이런 신호는 문제 지문에서 자주 보이는 단서예요. 관련 개념을 누르면 그 주제로 바로 이동해요. 한 조합씩 대표 유형을 직접 풀어 보면 가장 빨리 익혀져요.

정렬 + 그리디

  • 이런 신호 : "최대한 많이", "겹치지 않게", 시작과 끝 시간이 주어질 때
  • 접근 : 끝나는 시간(또는 알맞은 기준)으로 정렬한 뒤, 조건이 되면 욕심내어 차례로 선택해요.
  • 관련 개념 : 정렬, 그리디
  • 대표 유형 : 회의실 배정, 강의실 배정, 작업 스케줄링

이분 탐색 + 그리디 (파라메트릭 서치)

  • 이런 신호 : "가능한 최대/최소 값을 구하라", 값이 커질수록 조건이 한 방향으로만 바뀔 때
  • 접근 : 답 자체를 이분 탐색해요. 후보 값 하나를 정하고 "이 값이 가능한가?"를 그리디로 판정하며 범위를 좁혀요.
  • 관련 개념 : 이분 탐색, 그리디
  • 대표 유형 : 나무 자르기, 공유기 설치, 랜선 자르기, 예산

BFS + 비트마스크 상태

  • 이런 신호 : 미로인데 열쇠나 아이템을 모아야 하고, 같은 칸도 가진 것에 따라 다르게 취급될 때
  • 접근 : 위치만이 아니라 "가진 열쇠 집합"을 비트로 함께 상태에 넣어 (행, 열, 비트마스크)로 BFS 해요.
  • 관련 개념 : DFS와 BFS, 비트마스킹
  • 대표 유형 : 열쇠와 문, 아이템 수집 미로, 최소 이동

위상 정렬 + DP

  • 이런 신호 : 선후 관계(먼저 해야 하는 것)가 있고, 그 위에서 최장 시간이나 경로를 구할 때
  • 접근 : 위상 순서대로 노드를 처리하며, 그 순서로 DP 값을 누적해요(앞이 확정된 뒤 뒤를 계산).
  • 관련 개념 : 위상 정렬, 동적 계획법 (DP)
  • 대표 유형 : 게임 개발(빌드 시간), 선수과목 최장 경로, ACM Craft

다익스트라 + 상태 / 조건

  • 이런 신호 : 최단 경로인데 "특정 지점을 거쳐서", "벽을 한 번 부수고" 같은 추가 조건이 붙을 때
  • 접근 : 노드에 조건(거친 여부, 부순 횟수)을 더한 확장 상태로 다익스트라를 돌려요.
  • 관련 개념 : 최단 경로 (다익스트라), 비트마스킹
  • 대표 유형 : 경유지 최단 경로, 벽 부수고 이동, 최소 비용 구하기

트리 + DP (트리 DP)

  • 이런 신호 : 트리 구조에서 "인접하지 않게 고르기", 부분 트리마다 최적을 모아야 할 때
  • 접근 : 후위 순회로 자식의 결과를 먼저 구하고, 그걸 합쳐 부모의 답을 만들어요.
  • 관련 개념 : 트리 기본과 순회, DP 심화
  • 대표 유형 : 트리의 독립집합, 우수 마을, 사회망 서비스

좌표 압축 + 펜윅 트리

  • 이런 신호 : 값의 범위는 매우 큰데 개수는 적고, 구간 합이나 "앞에 나보다 큰 수의 개수"가 필요할 때
  • 접근 : 값을 순위로 압축한 뒤 펜윅 트리로 개수나 합을 누적하며 질의해요.
  • 관련 개념 : 좌표 압축, 세그먼트 트리와 펜윅 트리
  • 대표 유형 : 역순쌍(inversion) 세기, 구간 빈도, 정렬 거리

해시 + 투 포인터 / 슬라이딩 윈도우

  • 이런 신호 : "조건을 만족하는 가장 짧은/긴 구간", "서로 다른 원소 K개" 같은 연속 구간 문제
  • 접근 : 두 포인터로 구간을 늘이고 줄이며, 해시(딕셔너리)로 구간 안 원소의 개수나 등장을 관리해요.
  • 관련 개념 : 해시, 투 포인터와 누적 합
  • 대표 유형 : 부분합, 서로 다른 부분 문자열, 최소 윈도우

누적 합 + 이분 탐색

  • 이런 신호 : 구간 합을 빠르게 구해야 하고, 그 합이 단조롭게 늘어나(음수가 없어) 경계를 찾을 때
  • 접근 : 누적 합을 만든 뒤, 조건을 만족하는 위치를 이분 탐색으로 찾아요.
  • 관련 개념 : 투 포인터와 누적 합, 이분 탐색
  • 대표 유형 : 합이 기준 이상인 최소 길이, 구간 합 질의

완전 탐색 + 비트마스크 / 가지치기

  • 이런 신호 : 모든 경우를 따져야 하는데 원소가 적고(스무 개 안팎), 상태를 집합으로 표현할 수 있을 때
  • 접근 : 부분집합을 비트로 돌거나 백트래킹으로 만들되, 가망 없는 가지는 일찍 끊어 줄여요.
  • 관련 개념 : 완전 탐색, 비트마스킹
  • 대표 유형 : N-Queen, 부분집합 합, 외판원(TSP), 스도쿠

BFS + 시뮬레이션 (구현)

  • 이런 신호 : 격자에서 여러 지점이 동시에 퍼지거나, 시간이 흐르며 상태가 바뀌는 문제
  • 접근 : 시작점을 모두 큐에 먼저 넣고(다중 시작 BFS) 동시에 퍼뜨려요. 규칙은 구현으로 꼼꼼히 처리해요.
  • 관련 개념 : DFS와 BFS, 구현과 시뮬레이션
  • 대표 유형 : 토마토, 불!, 안전 영역, 인구 이동

정렬 + 유니온 파인드 (크루스칼)

  • 이런 신호 : "모든 지점을 최소 비용으로 연결", 간선과 가중치가 주어질 때
  • 접근 : 간선을 가중치 순으로 정렬하고, 사이클을 만들지 않는 간선만 유니온 파인드로 골라 이어요.
  • 관련 개념 : 정렬, 분리 집합 (Union-Find), 최소 신장 트리
  • 대표 유형 : 최소 신장 트리, 네트워크 연결, 도로 건설

모듈러 + 조합 / 거듭제곱

  • 이런 신호 : "경우의 수를 1000000007로 나눈 나머지", 답이 매우 커지는 세기 문제
  • 접근 : 곱셈마다 나머지를 취하고, 나눗셈이 필요하면 페르마 소정리로 역원을 곱해요. 큰 지수는 빠른 거듭제곱으로 처리해요.
  • 관련 개념 : 모듈러 연산, 조합론
  • 대표 유형 : 경우의 수 mod, 큰 이항 계수, 격자 경로 수
마무리 : 조합 문제가 막힐 때는 "이 지문에서 가장 강한 신호가 무엇인가"부터 찾아보세요. 신호 하나가 두 번째 개념을 자연스럽게 끌고 와요. 아래 유형별 추천 문제에서 실제 문제로 연습하면 좋아요.
